## AI Browser - 技术架构文档

版本：2.0.0 | 更新日期：2026-07-02

---

## 1. 项目概述

AI Browser 是一个基于 Chrome Extension Manifest V3 的浏览器扩展，采用纯 Vanilla JavaScript（ES Modules）开发，无构建步骤、无依赖打包工具。项目通过 Service Worker 作为中枢，协调 SidePanel、Popup、Content Scripts 和外部 Admin Server 之间的通信，实现 AI 对话、自主网页操作（Agent）、脚本管理等功能。

### 技术栈总览

| 层次 | 技术 | 说明 |
|------|------|------|
| 扩展框架 | Chrome Extension Manifest V3 | Service Worker 替代 Background Page |
| 编程语言 | Vanilla JavaScript (ES Modules) | 无 TypeScript，无编译 |
| 构建系统 | 无 | 直接加载源码到 Chrome |
| UI 渲染 | Vanilla DOM + Shadow DOM | 无框架依赖，Shadow DOM 隔离样式 |
| 样式方案 | 内联 CSS + CSS Custom Properties | 通过 JS 动态注入 |
| AI 后端 | OpenAI 兼容 API | 通过服务端代理 `/api/ai-proxy/chat` |
| 认证机制 | HMAC-SHA256 | AppKey + Timestamp 签名 |
| 数据存储 | chrome.storage.local / IndexedDB / localStorage | 三级存储策略 |
| 进程通信 | chrome.runtime (RPC + Port) / BroadcastChannel | 多种通信模式 |
| 默认模型 | deepseek-v4-pro | 可配置切换 |

### 构建与部署

项目无需构建，直接在 Chrome 扩展管理页面以"开发者模式"加载解压目录即可。更新时替换目录文件，Chrome 自动检测 manifest 变化并触发重载。

## 3. 模块详解

### 3.1 Background Service Worker

**入口文件：** `background/index.js`

Service Worker 是整个扩展的中枢，负责：
1. 实例化所有服务模块（依赖注入模式）
2. 注册消息路由（RPC 调度）
3. 管理 Port 流式连接
4. 监听系统事件（安装、闹钟、右键菜单、快捷键）

**初始化流程：**

```
index.js 加载
  → 实例化 ConfigService / StorageService（基础配置）
  → 实例化 AIService / ScriptService / ToolService（业务服务）
  → 实例化 ToolRecordingService / AgentResumeService（辅助服务）
  → 实例化 AgentService（核心引擎，注入依赖）
  → 实例化 TaskTemplateService / HumanInterventionService
  → 实例化 ScheduledTaskService（注入 navigate/inject/sendAgent 回调）
  → 注册 chrome.runtime.onMessage 监听
  → 注册 chrome.runtime.onConnect 监听
  → 注册 chrome.runtime.onInstalled 事件
```

**RPC 调度机制：**

SidePanel 和 Popup 通过统一的 `callService` 消息格式调用后台服务：

```javascript
// 请求格式
{ type: 'callService', service: 'configService', method: 'getAIConfig', args: [] }

// 响应格式
{ error: null, data: { model: 'deepseek-v4-pro', ... } }
// 或
{ error: 'Service not found: xxx', data: null }
```

Service Worker 通过 `services` 对象查找服务实例，使用 `Promise.resolve` 包装以兼容同步和异步方法。

### 3.2 Agent 引擎（核心模块）

Agent 引擎是产品最复杂的模块，由多个子模块协同工作：

```
AgentService (agent-service.js)
  ├── agent-runner.js       ← 主运行循环（LLM 调用、工具分发、阶段切换）
  ├── agent-dom-executor.js ← 15+ DOM 工具实现
  ├── agent-tool-builder.js ← 工具定义构建器（按阶段生成 tool schema）
  ├── agent-judge.js        ← 任务评判 & 经验记忆
  ├── agent-payload-utils.js← 结果截断与存储判断
  └── todo-scheduler.js     ← 三阶段待办调度引擎
```

#### Agent 生命周期

```
用户发送 Agent 指令
    │
    ▼
agentService.startAgent(port, userMessage, chatHistory)
    │
    ├── 检查是否已有运行中的 Agent（同 tabId），如有则中止旧任务
    ├── 创建运行上下文 ctx（状态、缓存、配置）
    ├── 初始化 TodoScheduler、PayloadStore、GlobalDataStore
    │
    ▼
runAgent(ctx) — agent-runner.js
    │
    ├── 1. 加载域名安全策略
    ├── 2. 读取 Agent 配置（maxRounds、enableJudge、debug）
    ├── 3. 清洗历史消息（移除尾部失败记录）
    ├── 4. 自动读取当前页面内容（extractPageContent）
    ├── 5. 自动搜索服务端工具库（关键词提取 + 意图扩展）
    ├── 6. 构建系统提示词（注入页面上下文和脚本信息）
    │
    ▼
主循环开始（while aiRequestCount < maxRounds）
    │
    ├── 检查中止信号、超时、调用次数上限
    ├── 同步 currentPhase（正常完成路径的阶段切换）
    ├── 生成收敛提示（70%/85% 预算阈值）
    ├── 注入待办进度上下文
    ├── 构建当前阶段的工具定义列表
    ├── 聚合系统消息注入
    │
    ├── API 请求（带重试、超时、fallback）
    │     │
    │     ├── 工具调用响应 → 逐个执行工具
    │     │     ├── 工具名称验证（幻觉拦截）
    │     │     ├── 执行工具（DOM/脚本/辅助）
    │     │     ├── 待办进度匹配与更新
    │     │     ├── 硬性规则检查（自动阶段切换）
    │     │     ├── PayloadStore 存储判断
    │     │     └── 工具录制记录
    │     │
    │     ├── 纯文本响应 → 流式输出并结束
    │     │
    │     └── 错误处理 → 重试 / fallback / 终止
    │
    ├── 消息上下文压缩（超过 40 条时）
    │
    └── finish_task → 输出结果 → 事后评判 → 结束
```

#### 三阶段执行模型

**Stage 1（页面探索）：** 使用 15+ DOM 工具直接在页面上操作。工具通过 `chrome.scripting.executeScript` 注入到页面的 MAIN world 执行。每个工具函数都是自包含的（闭包变量在序列化时会丢失，因此 `qsa` 辅助函数在每个工具内重新定义）。

DOM 工具清单：

| 工具名 | 功能 | 关键参数 |
|--------|------|---------|
| `read_page_content` | 读取页面标题、URL、正文 | - |
| `extract_content` | CSS 选择器提取内容（支持 `:contains` 伪类） | selector, multiple, limit, attributes |
| `click_element` | 点击元素 | selector, index |
| `fill_input` | 填写输入框（模拟 React/Vue 原生 setter） | selector, value, submit |
| `wait_for_element` | 等待元素出现 | selector, timeout |
| `navigate_to` | 导航到 URL（受域名策略限制） | url |
| `go_back` / `go_forward` | 浏览器前进/后退 | - |
| `scroll_page` | 滚动页面 | direction, amount |
| `hover_element` | 悬停元素 | selector, index |
| `select_dropdown` | 选择下拉框选项 | selector, value, by |
| `press_key` | 模拟按键 | key, selector |
| `find_text_on_page` | 页面文本搜索 | query, caseSensitive |
| `get_element_info` | 获取元素详细信息 | selector, limit, attributes |
| `get_interactive_elements` | 获取可交互元素列表 | selectorHint |
| `screenshot_visible` | 截取当前可见区域 | - |

**Stage 2（脚本处理）：** 调用服务端脚本批量处理数据。脚本通过 `ToolService.executeTool()` 注入执行，支持 precheck 前置校验、域名匹配、成功率排序。

**Stage 3（结果汇总）：** 仅暴露 `recall_data` 和 `finish_task` 两个工具，AI 汇总所有已收集数据并输出最终结果。

#### Todo 调度引擎 (TodoScheduler)

TodoScheduler 是 Agent 的进度管理核心，负责：

1. **待办模板与校验：** AI 创建三阶段待办列表后，系统执行严格校验：
   - 阶段工具合规性（Stage 1 禁止 `inject_script_*`，Stage 2 只允许 `inject_script_*` 和辅助工具）
   - 数据依赖合法性（`dataDependKeys` 必须引用之前待办的 `dataOutputKey`）
   - `dataOutputKey` 唯一性

2. **进度追踪：** 每次工具调用后匹配到当前待办，标记完成/失败，推进待办指针

3. **收敛控制：**
   - 70% 预算 → 软收敛提示
   - 85% 预算 → 紧急收敛提示

4. **硬性规则：**
   - Stage 1 连续 4 次无进展 → 自动切换 Stage 2
   - Stage 2 连续 3 次脚本失败 → 自动切换 Stage 3

#### 数据流设计

```
GlobalDataStore（跨阶段持久存储）
  │
  ├── set(key, data, source)     ← 待办完成时自动存储 dataOutputKey
  ├── get(key)                   ← recall_data 时查询
  ├── has(key)                   ← 依赖校验
  ├── getAllSummaries()          ← 阶段切换时生成摘要注入提示词
  └── getAllUrls()               ← 提取所有 URL 供脚本使用

PayloadStore（大结果截断与召回）
  │
  ├── 工具结果超过阈值 → 自动存储完整数据，发回摘要 + 存储 ID
  ├── recall_data(entry_id)     ← AI 按需召回完整数据
  └── getSummaryForFinish()     ← finish_task 时汇总所有存储
```

#### 上下文管理

- **消息压缩：** 当消息超过 40 条时，保留最近 60% 的消息，将早期消息按重要性分层压缩：
  - S 级：链接列表、脚本执行结果（保留 500 字符）
  - A 级：导航结果、create_todo/search_tools 结果（保留 120 字符）
  - B 级：其他工具结果（保留 100 字符，仅保留最后 6 条）
  - C 级：系统消息（去重，合并停滞提示）
- **孤立消息清理：** 压缩后移除没有对应 `tool_calls` 的孤立 `tool` 消息

#### 工具幻觉拦截

每次工具调用前，验证工具名称是否在当前阶段的允许工具列表中。不在列表中的调用被拦截并返回错误提示，防止 AI 调用不存在的工具。

#### 任务评判 (AgentJudge)

任务完成后（`finish_task`），可选启用 AI 评判器：
- 评估任务完成度（success / partial / failure）
- 输出评语
- 评判结果追加到聊天输出中

#### 断点恢复 (AgentResumeService)

支持 Agent 执行过程中的快照保存和恢复，当 SidePanel 断开重连时，Agent 可以继续执行而不丢失进度。

#### 人机交互 (HumanInterventionService)

当 Agent 遇到需要用户决策的情况时，通过回调向 SidePanel 发送介入请求，用户响应后 Agent 继续执行。

### 3.3 AI 服务

**文件：** `ai-service.js`

#### 流式通信协议

SidePanel 通过 Chrome Port 与 Background 建立流式连接：

```
SidePanel                          Background
   │                                   │
   ├── port = connect('ai-stream') ──→ │
   │                                   │
   ├── port.postMessage({              │
   │     type: 'streamStart',          │
   │     messages: [...],              │
   │     options: {...}                │
   │   }) ──────────────────────────→  │
   │                                   ├── fetch SSE API
   │   ←── streamChunk (逐字符) ──────┤
   │   ←── streamChunk ───────────────┤
   │   ←── streamDone ────────────────┤
   │                                   │
```

Agent 使用相同的 Port 模式，端口名为 `agent-stream`。

#### 请求签名机制

```javascript
// HMAC-SHA256 签名（纯 JS 实现，兼容 HTTP 页面）
const timestamp = Math.floor(Date.now() / 1000)
const signature = hmacSha256(appSecret, appKey + timestamp)
headers = {
  'X-App-Key': appKey,
  'X-Timestamp': timestamp,
  'X-Signature': signature,
}
```

#### 聊天历史管理

- 持久化到 `chrome.storage.local`
- 字符级截断（约 8000 字符上限）
- 最多保留 50 条消息
- Agent 模式下自动清理尾部失败记录

### 3.4 Content Scripts

**注入策略：**

| 脚本 | 注入时机 | 职责 |
|------|---------|------|
| `network-capture.js` | `document_start` | 拦截 XHR/Fetch |
| `index.js` | `document_idle` | UI 注入、内容提取 |

#### Shadow DOM 隔离

所有注入的 UI 元素（浮动按钮、划词工具栏、待办面板）都使用 Shadow DOM（`mode: 'closed'`）隔离样式，确保不受宿主页面 CSS 影响。

#### 浮动按钮系统

- 5 个按钮（工具箱、工具、待办、设置、AI）固定在页面右侧边缘
- MutationObserver 守护：如果按钮被页面脚本移除，自动重新注入
- 扩展上下文失效检测：提醒用户刷新页面

#### 划词工具栏

- 监听 `mouseup` 事件，检测选中文本长度 ≥ 3 字符
- 使用 `getBoundingClientRect()` 定位工具栏在选中文本上方
- 四个操作按钮：解释、翻译、改写、摘要

#### 智能表单填充

- MutationObserver 监听 DOM 变化，持续扫描新增表单
- 过滤条件：至少 2 个可填写字段，排除纯搜索框
- 使用原生 `HTMLInputElement.prototype.value` setter 赋值，兼容 React/Vue 受控组件
- WeakMap 避免重复注入按钮

#### 网络请求拦截 (network-capture.js)

- 在 `document_start` 阶段拦截 `fetch` 和 `XMLHttpRequest`
- 流式读取响应体（`ReadableStream`），上限 16KB
- 批量上报（定时器合并），减少消息频率
- 通过 `window.__aiBrowserGetCaptured` 暴露查询接口供 Agent 使用

#### Nonce 回调验证

```javascript
// Content Script 生成随机 Nonce
const nonce = Math.random().toString(36).slice(2) + Date.now().toString(36)
Object.defineProperty(window, '__AI_BROWSER_CB_NONCE__', {
  value: nonce,
  writable: false, configurable: false, enumerable: false  // 防枚举
})

// 注入脚本回调时必须携带 nonce
window.postMessage({ type: 'AI_BROWSER_CALLBACK', nonce, data })

// Content Script 校验 nonce 匹配后转发到 Background
```

### 3.5 Popup

**文件：** `popup/popup.js`, `popup/popup.html`

三标签页结构：AI 助手、脚本管理、设置。所有操作最终引导用户打开 SidePanel 以获得完整体验。Popup 通过 `chrome.storage.local` 的 `pendingMessage` 键向 SidePanel 传递待发送消息。

### 3.6 SidePanel

**文件：** `sidepanel/sidepanel.js`, `sidepanel/sidepanel.html`, `sidepanel/sidepanel.css`

#### 聊天界面

- Markdown 渲染（支持代码块、链接、列表）
- XSS 安全：使用 `textContent` 渲染，链接协议过滤（仅 http/https/mailto）
- 流式字符级输出动画

#### Agent 模式

- 步骤卡片渲染：工具名称、参数、执行状态、结果摘要
- 工具调用搜索结果的专用展示
- 阶段切换和进度更新

#### 文件上传

- 图片：转为 base64 Data URL，附加到消息中（需视觉模型支持）
- PDF：发送到服务端 `/api/ai-proxy/parse-pdf` 提取文本，附加到消息上下文
- 附件持久化：使用 `localStorage` 同步存取，关闭重开 SidePanel 后恢复

#### 功能面板 (feature-panels.js)

可拖动的浮动窗口，包含标签页：

- **执行图谱：** 三阶段流水线可视化
- **资源监控：** Token 使用量、API 调用次数统计
- **任务模板：** 模板管理和快速启动
- **工具录制：** 录制会话列表和详情
- **定时任务：** 任务 CRUD 和管理
- **调试日志：** Agent 调试信息实时展示

## 4. 通信机制

### 通信模式汇总

| 模式 | 协议 | 方向 | 用途 |
|------|------|------|------|
| RPC | `chrome.runtime.sendMessage` | SidePanel/Popup → Background | 服务方法调用（callService） |
| 流式 | `chrome.runtime.connect` (Port) | SidePanel ↔ Background | AI 对话流式输出、Agent 事件流 |
| Tab 消息 | `chrome.tabs.sendMessage` | Background → Content Script | 页面内容提取、待办更新、Agent 指令 |
| Runtime 消息 | `chrome.runtime.sendMessage` | Content Script → Background | 注入回调、表单填充请求、侧边栏控制 |
| BroadcastChannel | `BroadcastChannel` | 跨上下文 | 跨窗口/iframe 事件广播 |
| Storage 事件 | `chrome.storage.onChanged` | 全局 | 配置变更通知、待发消息传递 |

### RPC 消息协议

```javascript
// 请求
{
  type: 'callService',
  service: 'configService',    // 服务名（services 对象键名）
  method: 'getAIConfig',       // 方法名
  args: []                     // 参数数组
}

// 成功响应
{
  error: null,
  data: { model: 'deepseek-v4-pro', serverUrl: '...', ... }
}

// 错误响应
{
  error: 'Method not found: configService.xxx',
  data: null
}
```

### Port 流式消息协议

```javascript
// AI 对话流
port.name = 'ai-stream'
→ { type: 'streamStart', messages: [...], options: {...} }
← { type: 'streamChunk', content: '你' }
← { type: 'streamChunk', content: '好' }
← { type: 'streamDone' }
← { type: 'streamError', error: '...' }

// Agent 流
port.name = 'agent-stream'
→ { type: 'agentStart', userMessage: '...', chatHistory: [...] }
→ { type: 'agentAttach' }  // 重连已有 Agent
← { type: 'agentStart' }
← { type: 'agentStep', step: 1, toolName: 'extract_content', toolArgs: {...} }
← { type: 'agentStepResult', step: 1, toolName: '...', result: '...', done: false }
← { type: 'agentTodoUpdate', data: {...} }
← { type: 'agentStatus', text: '阶段1 第3轮' }
← { type: 'agentDebug', label: '...', detail: '...' }
← { type: 'streamChunk', content: '...' }  // finish_task 输出
← { type: 'streamDone' }
← { type: 'agentError', error: '...' }
```

### Port 弹性设计

Agent 运行过程中 SidePanel 可能断开（用户关闭侧边栏），此时：
- Agent 不终止，仅解除 Port 绑定（`detachPortByPort`）
- SidePanel 重新打开后，发送 `agentAttach` 消息重连
- Agent 检测到活跃任务后绑定新 Port，继续推送事件

## 5. 数据存储

### 三级存储策略

| 存储 | 用途 | 特点 |
|------|------|------|
| `chrome.storage.local` | 配置、聊天记录、注入回调、待发消息 | 异步 API，跨上下文可访问，5MB 配额 |

### chrome.storage.local 键值设计

|----|------|------|
| `aiConfig` | Object | AI 模型配置（model, serverUrl, maxTokens） |
| `appAuth` | Object | 认证信息（appKey, appSecret） |
| `syncConfig` | Object | 同步配置（syncInterval, serverUrl） |
| `chatHistory` | Array | 聊天历史记录 |

| `selectionToolsEnabled` | Boolean | 划词工具开关 |

### GlobalDataStore 跨阶段数据流

```
Stage 1: extract_content → dataOutputKey="links"
  └→ globalDataStore.set("links", [...], "s1-2")

Stage 2: inject_script_10(dataDependKeys=["links"])
  └→ globalDataStore.get("links") → 传入脚本参数
  └→ 脚本结果 → dataOutputKey="details"
  └→ globalDataStore.set("details", [...], "s2-1")

Stage 3: finish_task(dataDependKeys=["links","details"])
  └→ globalDataStore.getAllSummaries() → 注入提示词
```

### PayloadStore 大结果截断

```javascript
shouldStoreToPayload(result, funcName)
  → 判断结果是否超过截断阈值
  → 是：storeToPayload() 存储完整数据，返回摘要 + 存储 ID
  → 否：smartTruncateResult() 轻度截断

// AI 召回
recall_data(entry_id="p1") → payloadStore.query({ entry_id: "p1" })
```

## 6. 安全设计

### HMAC-SHA256 请求签名

所有到 Admin Server 的请求携带签名头：

```
X-App-Key: <appKey>
X-Timestamp: <unix_seconds>
X-Signature: HMAC-SHA256(appSecret, appKey + timestamp)
```

采用纯 JS 实现 HMAC-SHA256（非 `crypto.subtle`），兼容 HTTP 页面上下文。

### 域名安全策略 (DomainPolicy)

- 从配置加载允许的域名列表
- Agent 的 `navigate_to` 工具执行前检查目标 URL 是否在允许范围内
- 阻止导航到未授权域名，返回安全策略拦截提示

### URL 安全检查

```javascript
// 阻止危险协议
const BLOCKED_PROTOCOLS = ['javascript:', 'data:', 'vbscript:', 'file:', 'blob:']
// 阻止系统页面
const SYSTEM_URLS = ['chrome://', 'edge://', 'about:', 'chrome-extension://']
```

### XSS 防护

- **聊天渲染：** 所有文本内容使用 `textContent` 赋值，不直接 `innerHTML`
- **链接过滤：** 仅允许 `http://`、`https://`、`mailto:` 协议
- **HTML 转义：** 工具函数 `escapeHtml()` 处理特殊字符
- **待办面板：** 所有动态内容通过 DOM API + textContent 渲染

### Nonce 回调验证

注入脚本的回调消息通过随机 Nonce 验证来源，防止页面脚本伪造回调注入后台。Nonce 以 `enumerable: false` 定义在 window 上，防止页面脚本枚举发现。

### 脚本沙箱执行

ScriptSandboxService 提供隔离的脚本执行环境，限制脚本的访问范围。

## 8. 性能优化

### 上下文压缩策略

消息超过 40 条时触发四级压缩（详见 3.2 节），将早期操作历史压缩为一条摘要消息，大幅减少 Token 消耗。

### 消息截断与 LRU 缓存

- `LRUCache` 工具类用于页面内容读取缓存（同 URL 不重复提取）
- 工具结果通过 `smartTruncateResult` 轻度截断（保留关键信息）
- 大结果通过 `PayloadStore` 完整存储，仅发回摘要

### 网络捕获的批量上报

`network-capture.js` 使用定时器合并捕获的请求，批量发送到 Background，避免高频消息传递。

### 流式传输

AI 对话和 Agent 事件均采用流式传输（SSE over Port），逐字符推送到前端渲染，提供实时反馈体验。

### API 重试机制

Agent 的 LLM API 请求支持最多 2 次重试，包含超时控制（60 秒）和指数退避。当请求因内容过大返回 400/413 时，自动尝试不带 tools 参数的 fallback 请求。

## 9. 扩展性设计

### 添加新的 Agent 工具

1. 在 `agent-dom-executor.js` 的 `funcs` 对象中添加工具实现函数
2. 在 `argMap` 对象中定义参数映射
3. 在 `agent-tool-builder.js` 的 `buildPhase1Tools` 或 `buildPhase2Tools` 中注册工具 schema
4. 在 `todo-scheduler.js` 的 `STAGE1_TOOLS` 或 `STAGE2_TOOLS` 中添加白名单
5. 在 `agent-runner.js` 的工具分发逻辑中添加处理分支

### 添加新的服务模块

1. 在 `background/services/` 目录下创建服务文件
2. 在 `index.js` 中导入并实例化服务
3. 将服务实例添加到 `services` 对象中（键名即为 RPC 调用时的 service 名）
4. SidePanel/Popup 中通过 `callService('serviceName', 'methodName', ...args)` 调用

### 扩展功能面板

1. 在 `feature-panels.js` 中添加面板渲染函数
2. 在 `sidepanel.html` 的功能面板容器中添加面板 HTML 结构
3. 在 `initFeaturePanels()` 中注册面板初始化逻辑
