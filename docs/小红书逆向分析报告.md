# 小红书反爬逆向分析报告

> **分析周期**：2026-07-09 ~ 2026-07-10
> **目标**：逆向分析小红书搜索/详情接口签名机制，实现批量数据采集
> **核心结论**：
> - ✅ **x-s-common 算法已完全破解**：纯 Node.js 实现，957+ 次调用 0 签名错误
> - ✅ **x-s（XYS_ 格式）已完全破解**：mnsv2 VM 在纯 Node.js 中成功运行，动态生成 XYS_ 签名，无需浏览器/Electron
> - ✅ **mnsv2 VM 逆向完成**：ds.js + vendor-dynamic.js 模块 68316 + signV2Init() → 注册 mnsv2 全局函数
> - ✅ **独立签名模块 xys-sign-node.js**：完全脱离浏览器运行，提供 cookie 即可生成动态 XYS_ 签名
> - ✅ **XYS_ 不触发 300015**：与 XYW_ 不同，XYS_ 格式动态生成不会触发环境检测
> - ⚠️ **XYW_ 格式触发 300015**：`_webmsxyw` 生成的 XYW_ 签名即使通过浏览器 fetch 也触发"浏览器运行环境异常"
> - ⚠️ **测试账号已被封号**：采集期间被封，需新账号继续验证

## 二、签名机制解析

### 2.1 x-s（XYS_ 格式）— mnsv2 VM（✅ 已完全破解）

**这是本次逆向的核心成果**：mnsv2 字节码虚拟机在纯 Node.js 中成功运行，动态生成 XYS_ 签名。

#### 2.1.1 架构总览

```
┌─ ds.js (62KB) ───────────────────────────────────┐
│ 自解密 IIFE → 创建 _BHjFmfUMEtxhI 编译器基础设施   │
│ → 用 7557 字符字节码自举 → 生成基础函数            │
│ → 注册: _BHjFmfUMEtxhI, _dsf, _dsn, _dsl, __bc   │
└──────────────────────────────────────────────────┘
                        ↓
┌─ vendor-dynamic.8cd1891c.js (1.35MB, Webpack Chunk) ──┐
│ 模块 41439 (7.6KB): 加密工具 (MD5, CRC32, Base64)     │
│ 模块 68316 (577KB): _AUuXfEG27Xa3x 编译器 + 233081    │
│                      hex 字节码 + signV2Init()        │
│ → 调用 signV2Init() → 注册 mnsv2 全局函数              │
└──────────────────────────────────────────────────────┘
                        ↓
┌─ xys-sign-node.js (独立模块) ─────────────────────┐
│ init() → 加载 ds.js + vendor-dynamic.js           │
│        → 执行模块 68316 → 调用 signV2Init()       │
│ sign() → seccore_signv2 算法 → XYS_ 签名          │
└──────────────────────────────────────────────────┘
```

#### 2.1.2 关键模块定位

| 模块 ID | 大小 | 内容 | 作用 |
|---------|------|------|------|
| 41439 | 7.6KB | MD5, CRC32, 自定义 Base64, encodeUtf8 | 加密工具库 |
| **68316** | **577KB** | `_AUuXfEG27Xa3x` 编译器 + 233081 hex 字节码 + `signV2Init()` | **mnsv2 注册核心** |
| 12369 | 1.9KB | `seccore_signv2` 调用 `window.mnsv2` | 签名入口（运行时调用） |

> **关键发现**：之前的分析误认为 `_AUuXfEG27Xa3x` 在模块 12369 或 41439 中。通过模块边界扫描（大括号深度匹配），确认所有编译器代码都在模块 68316（vendor-dynamic.js 中最大的模块）。

#### 2.1.3 mnsv2 注册流程

```
1. 加载 ds.js
   → eval(dsCode)
   → 注册 _BHjFmfUMEtxhI, _dsf, _dsn, _dsl, __bc

2. 加载 vendor-dynamic.js (webpack chunk 格式)
   → 拦截 webpackChunkxhs_pc_web.push()
   → 收集 124 个模块到 webpackModules 对象

3. 执行模块 68316
   → webpackRequire('68316')
   → 模块内部定义 _AUuXfEG27Xa3x 编译器函数
   → 模块导出 { a: signV2Init }
   → ⚠️ signV2Init 不会自动调用！

4. 调用 signV2Init()
   → mod.a()
   → 内部执行 glb[_0xe762c0(0x73)]=function(...)  ← 注册 _AUuXfEG27Xa3x
   → 内部执行 glb['_AUuXfEG27Xa3x'](__$c, deps)   ← 编译字节码
   → mnsv2 全局函数注册完成
```

#### 2.1.4 seccore_signv2 算法

**源文件**：`vendor-dynamic.8cd1891c.js` 模块 12369 中的 `seccore_signv2(e, a)` 函数

```javascript
function seccoreSignV2(apiPath, body) {
  // 1. 构建签名输入
  let c = apiPath;
  if (body) c += typeof body === 'object' ? JSON.stringify(body) : body;

  // 2. MD5 哈希
  const u = md5Hex(c);        // W.Pu = MD5(c)
  const p = md5Hex(apiPath);  // W.Pu = MD5(apiPath)

  // 3. mnsv2 VM 签名
  const v = mnsv2(c, u, p);   // 返回 "mns0201_..." 签名

  // 4. 构建 payload
  const S = {
    x0: '4.3.7',         // 指纹版本
    x1: 'xhs-pc-web',    // 应用 ID
    x2: 'Windows',        // 平台
    x3: v,               // mnsv2 签名
    x4: body ? typeof body : '',  // 请求体类型
  };

  // 5. 编码
  const jsonStr = JSON.stringify(S);
  const utf8Bytes = encodeUtf8(jsonStr);
  const b64 = customBase64Encode(utf8Bytes);

  return { 'X-s': 'XYS_' + b64, 'X-t': String(Date.now()) };
}
```

#### 2.1.5 mnsv2 VM 结构

mnsv2 是一个两层嵌套 VM 签名字符串生成器：

| 属性 | 值 | 说明 |
|------|-----|------|
| `IIΙ` | 27188 | 字节码入口偏移（hex 字符索引） |
| `IΙI` | 26 | 字节码长度（hex 字符数 = 13 字节） |
| `IΙΙ` | Function | 字节码解释器（`_0x1233dd`，31126 字符） |
| `ΙII` | 1 | 调用计数器 |
| `ΙIΙ` | Object | 运行时数据表（含 $0 依赖数组） |

**字节码格式**：纯 hex 字符串（每 2 字符 = 1 字节），变长指令编码（类 Protobuf varint）

### 2.2 x-s（XYW_ 格式）— _webmsxyw（⚠️ 触发 300015，已弃用）

**状态**：算法已破解，Node.js 可生成，但动态 XYW_ 签名会触发 300015 环境检测。

**生成函数**：`_webmsxyw(apiPath, body)`

**XYW_ 格式结构**：
```
XYW_ + Base64({
  "signSvn": "56",
  "signType": "x2",
  "appId": "xhs-pc-web",
  "signVersion": "1",
  "payload": "<416字符 hex编码, 208字节>"
})
```

**300015 问题分析**：

| 尝试方案 | 结果 | 根因 |
|----------|------|------|
| Node.js 生成 XYW_ + 直接 HTTPS 请求 | 300015 | TLS 指纹不匹配 |
| 浏览器 executeJavaScript 生成 XYW_ + 浏览器 fetch | 300015 | executeJavaScript 环境与真实页面不同 |
| 补全所有请求头 (x-b3-traceid 等) + 浏览器 fetch | 300015 | 仍被检测 |
| 补全 x-s-common + x-rap-param + 浏览器 fetch | 300015 | 仍被检测 |

**结论**：XYW_ 格式的动态生成（无论 Node.js 还是浏览器 executeJavaScript）均会触发 300015。改用 XYS_ 格式后问题消失。

### 2.3 x-s-common 算法（✅ 已破解）

**源文件**：`vendor-dynamic.8cd1891c.js` 模块 41439

**生成流程**：
```
构建 payload 对象 en = {s0, s1, x0~x12}
  → JSON.stringify(en)
  → encodeUtf8(str) → byte array
  → b64Encode(bytes) → 自定义 Base64 编码
  → 结果即为 x-s-common 值
```

**Payload 结构**：
```javascript
const en = {
  s0: 5,              // 固定值
  s1: '',             // 空字符串
  x0: '4.3.7',        // 指纹版本
  x1: 'web',          // 平台标识
  x2: 'PC',           // 设备类型
  x3: 'xhs-pc-web',   // 应用 ID
  x4: '6.31.2',       // webBuild 版本
  x5: a1_cookie,      // a1 cookie 值
  x6: mnsToken,       // X-Mns 相关（可空）
  x7: xsSign,         // X-Sign 头（可空）
  x8: fingerprint,    // localStorage 指纹（可空）
  x9: tb(mnsToken + xsSign + fingerprint),  // CRC32 变体哈希
  x10: sigCount,      // 签名计数
  x11: 'normal',      // 固定值
  x12: dsl,           // window._dsl（可空）
};
```

**关键函数**：

| 函数 | 作用 |
|------|------|
| `encodeUtf8(str)` | `encodeURIComponent` → 字节数组 |
| `b64Encode(bytes)` | 自定义字母表 Base64 编码（同 XYS_ 的字母表） |
| `tb(data)` | CRC32 变体（多项式 0xedb88320，返回 `(-1 ^ crc) ^ polynomial`） |

**验证结果**：957+ 次 feed API 调用中 0 个 300011 签名错误。

## 五、风控检测分析

### 5.1 检测根因

| 检测因素 | 严重度 | 说明 |
|----------|--------|------|
| 静态 x-s 复用 400+ 次 | 🔴 高 | 同一签名被大量复用，真实浏览器每次生成唯一签名 |
| 请求总量 957 次 | 🔴 高 | 远超正常用户浏览量 |
| sigCount 线性增长 1→417 | 🟡 中 | x-s-common 中的 x10 字段暴露自动化行为 |
| 固定 3.5s 间隔 | 🟡 中 | 无随机抖动，请求模式可被检测 |
| 无浏览行为模拟 | 🟡 中 | 纯 API 调用，无页面浏览/滚动/停留 |

### 5.2 错误码含义

| 错误码 | 含义 | 处理方式 |
|--------|------|----------|
| -1 | API 路径/签名不匹配 | 需要对应 API 路径的独立 x-s |
| -100 | 登录已过期/Cookie 被标记 | 检查 cookie 有效性 |
| 300011 | 签名校验失败 / 账号异常 | 无"账号异常"→检查签名；有→停止采集 |
| 300012 | 签名过期 | 重新生成签名 |
| 300013 | 请求频繁 | 降低请求频率，等待恢复 |
| 300015 | 浏览器运行环境异常 | **使用 XYS_ 格式（不触发），弃用 XYW_** |
| 300031 | 笔记无法浏览 | 笔记已下架或被删除 |
| 461 | 请求被拒绝 | XYW_+XYS_ 格式混用，或环境检测不通过 |

### 5.3 300015 环境检测深度分析

300015 是最复杂的检测机制，通过多维度判断请求是否来自真实浏览器：

| 检测维度 | XYW_ 格式 | XYS_ 格式 |
|----------|-----------|-----------|
| TLS 指纹 | ⚠️ Node.js TLS 与浏览器不同 | ✅ 不检测（格式本身不触发） |
| executeJavaScript 环境 | ⚠️ 与真实页面上下文不同 | ✅ 不依赖浏览器执行 |
| 请求头完整性 | ⚠️ 需要全部匹配 | ✅ 不触发此检测 |
| x-s-common 存在性 | ⚠️ 必需 | ✅ 不触发此检测 |

**结论**：300015 检测针对的是 XYW_ 格式的生成过程（需要浏览器环境），而 XYS_ 格式通过 `seccore_signv2` 在 Node.js 中生成，完全不触发此检测。

## 七、mnsv2 VM 逆向细节

### 7.1 字节码解释器

**函数签名**：`_0x1233dd(bytecodeHex, offset, length, userArgs, dataTable, context, arg6, arg7)`

**核心辅助函数**：

```javascript
// 变长整数读取器（类似 Protobuf varint）
_0x36b043 = function(bytecode, pos) {
    var head = parseInt(bytecode[pos] + bytecode[pos+1], 16);
    if (head >> 7 == 0)   return [1, head];           // 7-bit
    if (head >> 6 == 2)   return [2, ((head & 0x3f) << 8) + lo];  // 14-bit
    if (head >> 6 == 3)   return [3, ((head & 0x3f) << 16) + ...]; // 22-bit
};
```

**已知 opcode**：

| Opcode | 含义 |
|--------|------|
| 0x40 | 不等于比较 |
| 0x3b | 创建 VM 函数（mnsv2 由此创建） |
| 0x9c9 | 迭代器相关 |
| 0x17e0 | 循环控制 |
| 0x1828 | 循环控制 |
