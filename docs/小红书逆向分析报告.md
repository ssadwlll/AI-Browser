# 小红书反爬逆向分析报告

> **分析周期**：2026-07-09
> **目标**：逆向分析小红书搜索接口签名机制，实现批量数据采集
> **结论**：客户端完全自动化绕过不可行，推荐半自动模式或 Headless 浏览器方案

## 一、反爬体系总览

小红书采用 **ACE (Anti-Crawler Engine)** 多层防御体系，由三大支柱构成：

```
┌─────────────────────────────────────────────────────────┐
│                    ACE 反爬体系                          │
├─────────────────────────────────────────────────────────┤
│  签名层          │ 设备指纹层        │ 行为检测层       │
│  seccore_signv2  │ xhsFingerprintV3  │ metrics_report  │
│  X-s / X-t       │ a1 cookie         │ 事件监听         │
│  mnsv2虚拟机     │ webId / gid       │ ACE事件注册      │
└─────────────────────────────────────────────────────────┘
```

| 防御层 | 机制 | 分析状态 |
|--------|------|----------|
| 签名层 | `seccore_signv2` + `mnsv2` 虚拟机 | ✅ 已完全解析 |
| 设备指纹层 | `xhsFingerprintV3` + a1 cookie | ✅ 已完全解析 |
| 行为检测层 | metrics_report + 事件监听 | ✅ 已分析 |

## 二、签名机制深度解析（完整流程）

### 2.1 核心签名函数

| 属性 | 值 |
|------|----|
| **主函数** | `seccore_signv2(url, body)` |
| **位置** | webpack chunk 模块 11436，约 17906 行 |
| **虚拟机** | `window.mnsv2(url_body_str, md5_url_body, md5_url)` |
| **虚拟机来源** | AS脚本 `https://fe-static.xhscdn.com/as/v1/3e44/public/04b29480233f4def5c875875b6bdc3b1.js` (151KB) |
| **输出格式** | `XYS_` + 自定义 Base64 编码 |

### 2.2 签名生成完整流程（伪代码）

```javascript
function seccore_signv2(url, body) {
    // Step 1: 构建签名数据 u
    let u = url;
    if (typeof body === 'object' && body !== null) {
        u = url + JSON.stringify(body);
    } else if (typeof body === 'string') {
        u = url + body;
    }
    // 否则 u = url（无 body 或 body 为空）

    // Step 2: 计算双重 MD5
    const m = MD5(u);           // url + body 的 MD5
    const w = MD5(url);         // 仅 url 的 MD5

    // Step 3: 核心签名（Sanji 虚拟机）
    const C = window.mnsv2(u, m, w);
    // 返回格式: "mns0301_" + 自定义 Base64（恰好 200 字符）
    // 包含随机性：相同输入每次调用返回不同值

    // Step 4: 组装 Payload JSON
    const payload = {
        x0: "4.3.7",                      // 指纹版本（来自 R.i8）
        x1: "xhs-pc-web",                 // 平台标识（window.xsecappid）
        x2: window.xsecplatform || "PC", // 设备类型
        x3: C,                            // mnsv2 签名结果（200 字符）
        x4: typeof body                   // body 参数类型
    };

    // Step 5: 最终编码
    const utf8Bytes = encodeUtf8(JSON.stringify(payload));
    const customB64 = b64Encode(utf8Bytes);  // 自定义字母表 Base64
    return "XYS_" + customB64;
}
```

### 2.3 关键加密参数详解

#### 2.3.1 MD5 哈希（webpack 模块 41439，导出名 K.Pu）

标准 MD5 实现，输入字符串，输出 32 位十六进制字符串。

#### 2.3.2 UTF8 编码（webpack 模块 41439，导出名 K.lz）

```javascript
function encodeUtf8(e) {
    var a = encodeURIComponent(e), s = [], u = 0;
    for (; u < a.length; u++) {
        var m = a.charAt(u);
        if ("%" === m) {
            var w = parseInt(a.charAt(u+1) + a.charAt(u+2), 16);
            s.push(w), u += 2;
        } else s.push(m.charCodeAt(0));
    }
    return s;  // 返回字节数组
}
```

#### 2.3.3 🔑 自定义 Base64（webpack 模块 41439，导出名 K.xE）

#### 2.3.4 🔴 mnsv2 虚拟机签名（核心）

| 属性 | 值 |
|------|----|
| **函数位置** | `window.mnsv2` |
| **脚本来源** | AS 脚本 151KB（Sanji 虚拟机） |
| **输入参数** | `(url_body_str, md5_url_body, md5_url)` |
| **输出格式** | `"mns0301_" + 自定义 Base64`（恰好 200 字符） |
| **执行耗时** | 13-16ms |
| **关键特性** | **不幂等**（相同输入每次返回不同值，内含计数器/时间戳/随机数） |

**特殊行为**：
- 当输入非真实 MD5 哈希时，返回占位符 `"mns0301_0"`
- 依赖 Sanji 虚拟机（webpack 模块 60621，约 360KB 混淆代码）
- 无法在 Node.js/Python 中独立重现

**调用链**：
```
window.mnsv2 → _0x31ad27 → Sanji 解释器执行字节码
```

### 2.4 Payload 结构解析

解码 `XYS_` 后的 Base64，得到 JSON：

```json
{
  "x0": "4.3.7",         // 指纹版本号（R.i8）
  "x1": "xhs-pc-web",    // 平台标识（window.xsecappid）
  "x2": "Windows",       // 设备类型（window.xsecplatform）
  "x3": "mns0301_abc...", // mnsv2 签名结果（200 字符）
  "x4": "object"         // body 参数类型（string/object/"")
```

**mnsv2 返回值结构**（200 字符）：
- 前缀：`mns0301_`
- 后缀：自定义 Base64 编码的加密数据（192 字节）
- 包含：输入参数 + 设备指纹 + 时间戳 + 随机因子

### 2.6 验证结果

**Replay 验证**：
使用正确格式的签名重放请求，服务器返回：


✅ **签名格式被接受**，HTTP 200（业务错误 -101 是因为缺少登录 Cookie，与签名无关）

## 三、设备指纹机制

### 3.1 a1 Cookie 分析

```
a1 值: 19f42e36256ncg8lexpiads5lvbkxeqio9faflufr50000994936
长度: 52 字符
字符集: 19f42e365ncg8lxpiadsvbkqour0 (28个唯一字符)
编码: Base64URL 兼容
末尾: 50000994936 (疑似服务端序号/分片标识)
```

**关键结论**：
- a1 由**服务端下发**（Set-Cookie），非客户端生成
- localStorage 中**无关联键**，完全服务端管理
- **一旦 a1 被标记为可疑环境，客户端无法清除标记**

### 3.2 关联标识

### 3.3 xhsFingerprintV3 指纹对象

| 方法 | 说明 |
|------|------|
| `getV18()` | 完整指纹字符串（核心，调用慢=每次重算） |

**关键发现**：
- 指纹在**脚本初始化时预收集到闭包**中

## 八、被检测原因深度分析

### 8.1 核心原因：a1 cookie 服务端标记

**证据链**：
1. a1 由服务端下发，基于首次请求的 IP/UA/环境特征生成
2. 一旦环境被判定为自动化，a1 被打上可疑标记
3. 后续所有请求携带被标记的 a1，都会被识别
4. 客户端无法修改/清除标记

### 8.2 排除的原因

- ❌ 鼠标轨迹问题：metrics_report 不含轨迹数据
- ❌ 指纹伪装失败：指纹检测已通过
- ❌ UA 问题：UA 已移除 Electron 标识

## 十、推荐方案

### 10.1 方案 A：Headless 浏览器（推荐）

使用 Puppeteer/Playwright 调用浏览器环境中的签名函数：

```javascript
// Puppeteer / Playwright
const page = await browser.newPage();
await page.goto('https://www.xiaohongshu.com/explore');

// 调用页内签名函数
const sign = await page.evaluate((url, body) => {
    // 自定义 Base64 字母表映射
    const CUSTOM_B64 = "ZmserbBoHQtNP+wOcza/LpngG8yJq42KWYj0DSfdikx3VT16IlUAFM97hECvuRX5";
    const STD_B64   = "ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz0123456789+/";
    const encodeMap = {};
    for (let i = 0; i < 64; i++) encodeMap[STD_B64[i]] = CUSTOM_B64[i];

    // 构建 u
    let u = url;
    const bodyType = typeof body;
    if (bodyType === 'object' && body !== null) {
        u += JSON.stringify(body);
    } else if (bodyType === 'string') {
        u += body;
    }

    // 计算 MD5（需要引入 MD5 实现）
    const m = md5(u);
    const w = md5(url);
    const C = window.mnsv2(u, m, w);

    const payload = JSON.stringify({
        x0: "4.3.7",
        x1: "xhs-pc-web",
        x2: window.xsecplatform || "PC",
        x3: C,
        x4: body ? (bodyType === 'object' ? "object" : bodyType === 'string' ? "string" : typeof body) : ""
    });

    // UTF8 编码
    const utf8bytes = new TextEncoder().encode(payload);

    // 标准 Base64 编码后转自定义字母表
    const stdB64 = btoa(String.fromCharCode(...utf8bytes));
    const customB64 = stdB64.split('').map(c => encodeMap[c] || c).join('');

    return {
        'X-s': 'XYS_' + customB64,
        'X-t': String(Date.now())
    };
}, '/api/sns/web/v1/feed', {source_note_id:"xxx",num:20,cursor:""});

// 使用签名
const resp = await fetch('https://edith.xiaohongshu.com/api/sns/web/v1/feed', {
    headers: {
        ...sign,
        'Cookie': 'your_cookies',
        'User-Agent': '...'
    }
});
```

**优势**：
- ✅ 完整复用浏览器环境签名
- ✅ mnsv2 自动执行（无需解析 360KB 混淆代码）
- ✅ 自定义 Base64 已解析，可直接替换

### 10.2 方案 B：半自动模式

**工作流程**：
1. 人工在 ai-browser 中浏览小红书搜索页
2. 人工点击笔记卡片打开弹窗
3. 按 `F12` 或用 ai-browser 的逆向分析工具，执行 SSR 提取脚本
4. 脚本只读取 `__INITIAL_STATE__.note.noteDetailMap`，不触发任何请求
5. 数据存入 localStorage 或导出

**优势**：
- ✅ 不触发 API 风控（无直接 API 调用）
- ✅ 不触发行为检测（人工浏览行为真实）
- ✅ a1 不会被标记（正常浏览模式）

### 10.3 方案 C：代理服务注入

部署本地代理服务，在浏览器中注入脚本，Hook `XMLHttpRequest.prototype.send` 和 `fetch`，自动调用 `seccore_signv2` 添加签名。

## 十三、总结

小红书签名机制已完全解析，核心算法流程清晰：

1. **签名流程**：`url+body` → MD5 → `mnsv2` 虚拟机 → Payload JSON → 自定义 Base64
2. **自定义 Base64**：字母表已解析，可直接替换
3. **mnsv2 虚拟机**：依赖 360KB Sanji 解释器，无法脱离浏览器环境

**推荐方案**：
- **首选**：Headless 浏览器（Puppeteer/Playwright）复用 `window.mnsv2`
- **次选**：半自动模式（人工浏览 + SSR 提取）

**最终结论**：签名算法已解析，可通过 Headless 浏览器方案实现自动化，但 a1 标记仍是风控瓶颈。
