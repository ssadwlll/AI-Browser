# 小红书反爬逆向分析报告

> **分析周期**：2026-07-09 ~ 2026-07-10
> **目标**：逆向分析小红书搜索接口签名机制，实现批量数据采集
> **核心结论**：
> - ✅ **x-s-common 算法已完全破解**：从 `vendor-dynamic.8cd1891c.js` 模块 41439 提取生成算法，纯 Node.js 实现 `xs-common-node.js`，动态签名通过真实 API 校验（957+ 次调用，0 个 300011 签名错误）
> - ✅ **x-s（XYW_ 格式）已破解**：`webmsxyw-node.js` 在 Node.js 中动态生成 XYW_ 格式签名，每次请求唯一，绑定 API 路径 + 请求体
> - ✅ **双动态签名方案已集成**：每次 feed 请求动态生成 x-s（XYW_）+ x-s-common，避免静态签名复用被检测
> - ✅ **搜索批量采集完全可用**：v2 API + 全部浏览器请求头 + XYS_ 静态签名复用，20关键词×3页=1332条笔记，0错误
> - ✅ **详情批量采集已验证**：16关键词，957条笔记详情成功采集
> - ⚠️ **风控检测为行为层面**（非签名失败）：957次请求后触发"账号异常"（300011+msg），根因是静态 x-s 复用 400+ 次 + 请求量过大
> - ✅ **Cookie 标记根因已查明**：使用 edith v1 API + 缺少 x-b3-traceid 等请求头 + 缺少 message_id 字段

| 防御层 | 机制 | 破解状态 | 说明 |
|--------|------|----------|------|
| 请求头检测层 | x-b3-traceid, x-rap-param, x-xray-traceid | ✅ **已破解** | 缺少会导致 cookie 被标记；x-rap-param 可静态复用 |

## 二、签名机制解析

### 2.1 x-s-common 算法（✅ 已破解）

**源文件**：`vendor-dynamic.8cd1891c.js`（webpack 模块 41439）

**生成流程**：
```
构建 payload 对象 en = {s0, s1, x0~x12}
  → JSON.stringify(en)
  → encodeUtf8(str) → byte array
  → b64Encode(bytes) → 自定义 Base64 编码
  → 结果即为 x-s-common 值
```

**Payload 结构**：
```javascript
const en = {
  s0: 5,              // 固定值
  s1: '',             // 空字符串
  x0: '4.3.7',        // 指纹版本
  x1: 'web',          // 平台标识
  x2: 'PC',           // 设备类型
  x3: 'xhs-pc-web',   // 应用 ID
  x4: '6.31.2',       // webBuild 版本
  x5: a1_cookie,      // a1 cookie 值
  x6: mnsToken,       // X-Mns 相关（可空）
  x7: xsSign,         // X-Sign 头（可空）
  x8: fingerprint,    // localStorage 指纹（可空）
  x9: tb(mnsToken + xsSign + fingerprint),  // CRC32 变体哈希
  x10: sigCount,      // 签名计数
  x11: 'normal',      // 固定值
  x12: dsl,           // window._dsl（可空）
};
```

**关键函数**：

| 函数 | 作用 |
|------|------|
| `encodeUtf8(str)` | `encodeURIComponent` → 字节数组 |
| `b64Encode(bytes)` | 自定义字母表 Base64 编码 |
| `tb(data)` | CRC32 变体（多项式 0xedb88320，返回 `(-1 ^ crc) ^ polynomial`） |

**自定义 Base64 字母表**：
```
自定义: ZmserbBoHQtNP+wOcza/LpngG8yJq42KWYj0DSfdikx3VT16IlUAFM97hECvuRX5
标准:   ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz0123456789+/
```

**验证结果**（2026-07-10）：

| 测试 | x-s-common 来源 | HTTP | code | 结论 |
|------|----------------|------|------|------|
| A | 浏览器真实签名 | 461 | 300013 | 限流（非签名失败） |
| B | xs-common-node.js 动态生成 | 200 | 300013 | 限流（非签名失败） |

- 两个测试均未返回 300011（签名校验失败），说明动态生成的 x-s-common 通过服务端验证
- 957+ 次 feed API 调用中 0 个 300011 签名错误

### 2.2 x-s（XYW_ 格式）算法（✅ 已破解）

**生成函数**：`_webmsxyw(apiPath, body)`

**XYW_ 格式结构**：
```
XYW_ + Base64({
  "signSvn": "56",
  "signType": "x2",
  "appId": "xhs-pc-web",
  "signVersion": "1",
  "payload": "<416字符 hex编码, 208字节>"
})
```

**脚本依赖**：

| 脚本 | 来源 | 作用 | 大小 |
|------|------|------|------|
| ds.js | `as.xiaohongshu.com/api/sec/v1/ds` | 创建运行时（`_BHjFmfUMEtxhI`） | ~60KB |
| sign.js | signSvn 配置中的 `signUrl` | 定义 `_webmsxyw` 函数 | ~148KB |

**签名特性**：
- 绑定 API 路径（不同路径产生不同 payload）
- 绑定请求体（不同 body 产生不同签名）
- 非幂等（相同输入每次结果不同，前192字节确定，后16字节随机）
- 性能：~70ms/次

**signSvn 配置**（`localStorage.sdt_source_storage_key`）：
```json
{
  "signUrl": "https://fe-static.xhscdn.com/as/v1/f218/a15/public/04b29480233f4def5c875875b6bdc3b1.js",
  "signVersion": "1", "signType": "x2", "signSvn": "56",
  "commonPatch": ["/api/sns/web/v1/feed", "/api/sns/web/v1/note/like", ...],
  "desVersion": "2", "validate": true
}
```

**commonPatch 列表**（决定哪些 API 需要 x-s-common）：
- `/api/sns/web/v1/feed` — 笔记详情 ✅ 在列表中
- `/api/sns/web/v1/search/notes` — 搜索 ❌ 不在列表中

### 2.3 x-s（XYS_ 格式，旧版）

| 属性 | 值 |
|------|-----|
| 生成函数 | `mnsv2(apiPath)` — 单参数 |
| 输出 | `XYS_` + 自定义 Base64 编码的 JSON |
| Payload | `{x0:"4.3.7", x1:"xhs-pc-web", x2:"PC", x3:"mns0301_...", x4:"string"}` |
| 路径绑定 | ✅ 是 |
| 请求体绑定 | ❌ 不绑定（可跨关键词/页码复用） |
| Node.js 运行 | ❌ vm.js 加载失败（ds.js 可运行） |
| 当前用途 | 搜索 API 静态签名复用（请求量低，3次/关键词） |

**XYS_ vs XYW_ 对比**：

| 维度 | XYS_（旧） | XYW_（新） |
|------|-----------|-----------|
| 生成函数 | `mnsv2(apiPath)` | `_webmsxyw(apiPath, body)` |
| 编码方式 | 自定义 Base64 | 标准 Base64 |
| 请求体绑定 | ❌ | ✅ |
| Node.js 运行 | ❌ vm.js 不兼容 | ✅ ds.js + sign.js |
| 适用场景 | 搜索静态复用 | 详情动态生成 |

## 五、风控检测分析

### 5.1 检测根因

第二批采集 957 次请求后触发 300011+"账号异常"，属于**行为层面风控**（非签名验证失败）。

| 检测因素 | 严重度 | 说明 |
|----------|--------|------|
| 静态 x-s 复用 400+ 次 | 🔴 高 | 同一签名被大量复用，真实浏览器每次生成唯一签名 |
| 请求总量 957 次 | 🔴 高 | 远超正常用户浏览量（20-40倍） |
| sigCount 线性增长 1→417 | 🟡 中 | x-s-common 中的 x10 字段暴露自动化行为 |
| 固定 3.5s 间隔 | 🟡 中 | 无随机抖动，请求模式可被检测 |
| 无浏览行为模拟 | 🟡 中 | 纯 API 调用，无页面浏览/滚动/停留 |

### 5.2 300011 错误码双重含义

| 场景 | msg 内容 | 含义 | 处理 |
|------|----------|------|------|
| 签名无效 | 无"账号异常" | 签名校验失败 | 检查 x-s/x-s-common 生成 |
| 行为风控 | 包含"账号异常" | 账号被标记 | 停止采集，等待恢复或换号 |

### 5.3 已实施的防风控措施

| 措施 | 说明 | 状态 |
|------|------|------|
| 动态 x-s 生成 | webmsxyw-node.js 每次请求生成唯一 XYW_ 签名 | ✅ 已集成 |
| 动态 x-s-common | xs-common-node.js 每次请求动态生成 | ✅ 已集成 |
| 随机延迟 | 详情间隔 4-7s 随机，关键词间 15-30s 随机 | ✅ 已实现 |
| sigCount 循环 | 1-30 循环增长，到 30 后随机重置 | ✅ 已实现 |
| 单次会话限制 | 每次最多 5 个关键词 | ✅ 已实现 |
| 账号异常检测 | 300011+"账号异常" 自动停止 | ✅ 已实现 |
| 断点续采 | START_INDEX 环境变量支持从指定关键词继续 | ✅ 已实现 |

## 七、错误码参考

| 错误码 | 含义 | 处理方式 |
|--------|------|----------|
| 0 | 成功 | - |
| -1 | API 路径/签名不匹配 | 需要对应 API 路径的独立 x-s |
| -100 | 登录已过期/Cookie 被标记 | 检查是否使用了 v1 API 或缺少必需请求头 |
| 300011 | 签名校验失败 / 账号异常 | 无"账号异常"→检查签名；有"账号异常"→停止采集 |
| 300012 | 签名过期 | 重新抓取/生成签名 |
| 300013 | 请求频繁 | 降低请求频率，等待恢复 |
| 300031 | 笔记无法浏览 | 笔记已下架或被删除 |
| 461 | 请求被拒绝 | XYW_+XYS_ 格式混用，或环境检测不通过 |

### 8.1 WASM 加密层（2025 新版 mns0101_）

2025年7月小红书升级算法 [$TRAE_REF](https://blog.csdn.net/gaoliang2021/article/details/149666887)，引入 WASM 加密：

| 维度 | 旧版（mns0301_） | 新版（mns0101_） |
|------|------------------|------------------|
| 执行引擎 | Sanji VM（360KB JS） | WASM 模块 |
| 输入 | url + body → MD5 | 124 字节数组 |
| 破解难度 | 高 | 极高 |

**124 字节数组结构**：
```
[4字节常量 119,104,96,41]
+ [4字节随机数派生]
+ [8字节时间戳派生]
+ [4字节计数器 74,3,0,0]
+ [4字节属性检测 12,5,0,0]
+ [4字节API路径长度]
+ [8字节API路径MD5前缀]
+ [a1长度 + a1字符编码]
+ [appId长度 + "xhs-pc-web"编码]
+ [16字节常量尾部]
→ 与常量KEY逐位异或 → WASM加密 → "mns0101_" + 结果
```
