# AI Browser — 产品技术说明书

## 一、产品概述

**AI Browser** 是一款 AI 驱动的 Chrome 浏览器扩展（Manifest V3），版本 2.0.0。它将大语言模型（LLM）深度嵌入浏览器，提供**智能对话、页面分析、划词工具、脚本自动化、Agent 自主决策**五大核心能力。支持多模型切换、图片/PDF 附件、工具函数调用（Function Calling），可在任意网页上通过侧边栏或浮动按钮即时调用。

### 3.3 通信机制

```
SidePanel ←──→ Service Worker ←──→ Content Script
   (RPC)         (Port/msg)          (msg)
```

- **SidePanel → Service Worker**：通过 `chrome.runtime.sendMessage` 进行 RPC 调用，JSON 序列化 (`{ service, method, args }`)
- **Service Worker → SidePanel**：通过 `chrome.runtime.connect` 建立 `ai-stream`/`agent-stream` Port 长连接，进行流式推送
- **Service Worker → Content Script**：通过 `chrome.tabs.sendMessage` 发送 DOM 操作指令，页面通过 `window.postMessage` 回传脚本执行结果

## 四、Agent 自主决策代码执行详解

### 4.1 执行总览

Agent 模式的核心入口是 `AgentService.run()` 方法（[agent-service.js](file:///c:/phpstudy_pro/WWW/qwen3/ai-browser/chrome-extension/background/services/agent-service.js) 第 660 行）。整体是一个 **while 循环**，每一轮：

1. 构建工具定义（Function Calling Schema）
2. 调用 LLM API（携带消息历史 + 工具列表）
3. 解析 LLM 返回的 `tool_calls`
4. 逐个执行工具（DOM 操作 / 脚本注入 / 搜索）
5. 将工具结果反馈给 LLM
6. 重复直到 `finish_task` 或触发终止条件

### 4.4 while 循环单轮详解

每一轮循环执行以下步骤：

#### Step 1：前置检查（3 个硬上限）

```javascript
// 1) 全局超时检查
if (Date.now() - startTime > this.TIMEOUT_MS) { /* 终止 */ }

// 2) 工具调用次数检查
if (totalToolCalls >= this.MAX_TOOL_CALLS) { /* 终止 */ }

// 3) aiRequestCount++ 后由 while 条件检查
//    不会显式判断，while(aiRequestCount < maxRounds) 自动退出
```

#### Step 2：步骤预算通知（70% 阈值）

```javascript
const budgetRatio = aiRequestCount / maxRounds
if (budgetRatio >= 0.7 && budgetRatio < 0.75 && !_budgetWarned) {
  // 向 messages 注入 system 消息，提醒 AI 收尾
  messages.push({
    role: 'system',
    content: `注意：已使用 ${aiRequestCount}/${maxRounds} 轮...`
  })
}
```

#### Step 3：构建工具定义列表

```javascript
const tools = this.buildToolDefinitions(userMessage, searchResults, currentPageUrl)
```

工具列表包含两类：
- **固定核心工具**（始终可用）：`search_tools`、`read_page_content`、`click_element`、`fill_input`、`wait_for_element`、`finish_task`
- **动态脚本工具**（按域名过滤）：从 `searchResults` 中筛选，最多 6 个 `inject_script_*` 工具

#### Step 4：调用 LLM API（含重试）

```javascript
const body = {
  model: config.model,
  messages,
  temperature: 0.3,     // Agent 模式用低温，减少随机性
  max_tokens: 8192,     // 至少 8192，确保 tool_calls 不被截断
  tools,
  tool_choice: 'auto',
}
```

API 重试策略（3 种场景）：
- **429 (限流) / 5xx**：最多重试 2 次，退避 1s→2s
- **超时 (AbortError)**：同上
- **400 (Bad Request)**：降级为不带 `tools` 的普通请求重试（兼容不支持 FC 的模型）
- **其他 4xx**（如 401 鉴权失败）：不重试，直接抛错

#### Step 5：解析 AI 响应

```
LLM 响应
  ├── 有 tool_calls → 进入工具执行循环
  └── 无 tool_calls（纯文本）
       ├── 检查 _idleTextCount >= 2 → 强制终止（AI 在无效循环）
       └── 否则流式输出文本 → return
```

#### Step 6：逐个执行 tool_calls

```javascript
let shouldTerminateSequence = false  // 动作终止标志

for (const toolCall of msg.tool_calls) {
  if (shouldTerminateSequence) break  // 上一操作触发终止，跳过后续
  // ... 按 funcName 分派执行 ...
}
```

**工具执行分派**（按 `funcName` 路由）：

| funcName | 执行方式 | 关键逻辑 |
|----------|---------|---------|
| `finish_task` | 流式输出 → return | `shouldTerminateSequence = true` |
| `search_tools` | `toolService.search()` | 缓存结果到 `searchResults` 数组 |
| `read_page_content` | `pageService.getContent()` | 提取页面标题/URL/正文 |
| `inject_script_*` | `toolService.executeTool()` | 前置检查 (precheck) → 注入执行 → 经验记忆记录 |
| DOM 工具 (`click_element`/`fill_input`/`navigate_to` 等) | `executeDOMTool()` | 带 30s 超时的 `Promise.race` |

#### Step 7：边界约束检查（每轮工具执行后）

```
工具执行完成后，依次运行：
  ├── LoopDetector.record()       ← 记录动作 (name + selector + url)
  ├── LoopDetector.recordPageState() ← 记录页面指纹 (URL + 元素数量)
  ├── LoopDetector.getNudge()     ← 获取分级提醒
  │     ├── 5次重复 → 温和提示
  │     ├── 8次重复 → 警告
  │     └── 12次重复 → 强烈建议
  │     └── 页面5步无变化 → 页面停滞警告
  ├── 注入 nudge 到 messages     (如果有提醒)
  ├── 无进展检测:
  │     ├── ok===false → 无进展
  │     ├── search_tools → 不重置计数
  │     └── 有有效内容 → 有进展，重置计数
  └── consecutiveFailCount >= 5 → 强制终止
```

#### Step 8：结果处理与截断

```javascript
// 智能截断：保留 JSON 结构
const truncatedResult = this._smartTruncateResult(toolResult)
// 数组 → 保留前10条 + 总数
// 对象 → 截断超长字符串字段 (>500)
// 纯文本 → 一刀切截断 + 标记

  role: 'tool',
  tool_call_id: toolCall.id,

```

## 五、AI 能力边界约束体系详解

AI 的不可预测性源自三个维度：**空间维度**（能访问什么）、**时间维度**（能执行多久）、**行为维度**（会做什么）。约束体系从这三维切入，采用 4 层纵深防御。

### 5.1 空间维度的约束

#### 5.1.1 域名白名单/黑名单（L2）

**配置位置**：`ConfigService → getAgentConfig()` 返回的 `agentConfig`

**配置格式**：
```json
{
  "allowedDomains": ["*.taobao.com", "*.jd.com"],
  "prohibitedDomains": ["*.gov.cn", "*.bank.com"],
  "blockIPAddresses": true
}
```

**生效机制**：

1. **加载时机**：每次 `run()` 启动时调用 `_loadDomainPolicy()`，从配置中读取策略并缓存到实例属性
2. **未设置策略**：`_allowedDomains` 为 `null`，`_isUrlAllowed()` 直接返回 `true`——全部放行
3. **白名单优先**：如果 `allowedDomains` 有值，只允许匹配的域名
4. **黑名单兜底**：如果只设置了 `prohibitedDomains`，禁止匹配的域名
5. **自动匹配 www 变体**：`example.com` 也匹配 `www.example.com`，反之亦然

**拦截点**（仅 `navigate_to` 工具）：

```javascript
// agent-service.js 第 1108 行
if (funcName === 'navigate_to' && !this._isUrlAllowed(funcArgs.url)) {
  // 返回错误给 AI，不执行实际导航
  toolResult = '导航被安全策略阻止...'
  // 不设置 shouldTerminateSequence，后续工具可以继续执行
}
```

**匹配算法**（`_matchDomain`）：

```
输入: hostname="www.example.com", pattern="*.example.com"
  → h = "www.example.com", p = "*.example.com"
  → p.startsWith("*.") → domainPart = "example.com"
  → h.endsWith(".example.com") → true ✓

输入: hostname="example.com", pattern="example.com"
  → h="example.com", p="example.com"
  → h === p → true ✓

输入: hostname="www.example.com", pattern="example.com"
  → h === "www." + p → true ✓  (自动 WWW 变体)
```

#### 5.1.2 脚本域名过滤（L2）

在 `buildToolDefinitions()` 中，每个 `inject_script_*` 工具会根据脚本的 `urlPattern` 字段与当前页面 URL 做匹配：

```javascript
// 第 128-134 行
for (const s of (searchResults || []).slice(0, 12)) {
  if (s.urlPattern && s.urlPattern !== '*' && currentPageUrl) {
    if (!this._matchUrlToDomain(currentPageUrl, s.urlPattern)) {
      console.log('脚本域名不匹配，跳过:', s.name)
      continue  // ← 不匹配的脚本根本不会暴露给 AI
    }
  }
  // ... 添加到工具列表
}
```

**关键设计**：遍历脚本时跳过不匹配的（而非暴露后再让 AI 判断），从源头缩小 AI 的工具选择空间。

**匹配算法**（`_matchUrlToDomain`）：
- `urlPattern` 为 `*` 或空 → 全部匹配
- 支持逗号分隔的多模式：`"https://*.taobao.com/*,https://*.tmall.com/*"`
- 支持 `*` 通配符和 `?` 单字符通配符

#### 5.1.3 系统页面拦截（L1）

// _getTargetTab() 第 651 行
if (url.startsWith('chrome://') || url.startsWith('chrome-extension://')
    || url.startsWith('edge://') || url.startsWith('about:')) {
  return null  // 系统页面 → 拒绝执行任何工具

### 5.3 行为维度的约束

#### 5.3.1 ActionLoopDetector（L3）——「动作模式检测」

这是最复杂的约束机制，位于 [agent-service.js#L1-L56](file:///c:/phpstudy_pro/WWW/qwen3/ai-browser/chrome-extension/background/services/agent-service.js#L1-L56)。

**数据结构**：
```javascript
{
  windowSize: 15,            // 滑动窗口：最近 15 个动作
  recentActions: [],         // [{key: "click|.btn||", name, url}]
  consecutiveStagnant: 0,    // 连续页面无变化次数
  _lastPageState: null,      // "url|elementCount" 指纹
}
```

**动作 Key 生成**（`record()`）：
```javascript
key = `${name}|${selector}|${navUrl}`
// 例如: "click_element|.login-btn||"
//       "navigate_to||https://example.com|"
//       "inject_script_5|||"
```

**检测维度**：

| 维度 | 算法 | 阈值 |
|------|------|------|
| 动作级重复 | 统计最近 15 个 action key 的频率，取最大值 | ≥5 次: 提示 / ≥8 次: 注意 / ≥12 次: 严重警告 |
| 页面停滞 | 比较连续的页面指纹 `url\|elementCount` | ≥5 步无变化: 提醒 |
| DOM 操作无效 | 连续停滞 + click 重复 → 暗示选择器可能错误 | 组合判断 |

**提醒注入**：
```javascript
const nudge = loopDetector.getNudge()
if (nudge) {
  messages.push({ role: 'system', content: nudge })
  // 作为 system 消息注入到 LLM 上下文中，AI 能"看到"并调整策略
}
```

与 `consecutiveFailCount` 的关系：
- `consecutiveFailCount`：基于**工具执行结果**（ok===false 或无有效内容）
- `ActionLoopDetector`：基于**动作模式**（重复频率、页面变化）
- 两者互补：前者检测"工具是否成功"，后者检测"AI 是否在打转"

#### 5.3.2 连续无进展终止（L1）

```
MAX_CONSECUTIVE_FAILS = 5
```

每轮工具执行后，通过结构化解析判断是否有进展：

| 条件 | 判定 |
|------|------|
| `parsed.ok === false` | 无进展 |
| `funcName === 'search_tools'` | 不计入进展（搜到不等于任务推进） |
| `ok===true` 或 `ok===undefined` 且有 `result/content/title` | 有进展 |
| JSON 解析失败 | 无进展 |

触发阈值时，流式输出错误信息并终止：

```javascript
if (consecutiveFailCount >= MAX_CONSECUTIVE_FAILS) {
  for (const char of stopMsg) {
    this.postToUI(tabId, { type: 'streamChunk', content: char })
    await new Promise(r => setTimeout(r, 10))  // 逐字流式输出
  }
  return
}
```

#### 5.3.3 纯文本空闲检测（L2）

```
MAX_IDLE_TEXT = 2
```

AI 连续返回纯文本（不含 tool_calls）超过 2 轮 → 判定为陷入无效对话循环，强制终止。

```javascript
// 每次有 tool_calls 时重置
if (msg.tool_calls && msg.tool_calls.length > 0) {
  _idleTextCount = 0   // ← 重置
}

// 每次无 tool_calls 时累加
} else {
  _idleTextCount++
  if (_idleTextCount >= MAX_IDLE_TEXT) {
    // 强制终止："AI连续回复纯文本未使用工具"
  }
}
```

#### 5.3.4 步骤预算通知（L3）——「让 AI 感知自身生命周期」

```
触发阈值：70% 的 maxRounds
触发次数：仅一次（_budgetWarned 标志防止重复）
```

注入的提示词：
```
注意：已使用 6/8 轮 (75%)。请在剩余 2 轮内完成核心任务。
优先处理高价值项目，及时保存结果，必要时提前调用 finish_task。
```

这不是硬阻断，而是**软提醒**——让 AI 知道资源有限并调整策略（优先做高价值子任务，放弃低优先级部分）。

#### 5.3.5 动作终止标志（L2）

```javascript
let shouldTerminateSequence = false

// navigate_to 或 go_back 执行后
if (['navigate_to', 'go_back'].includes(funcName) && !toolResult.includes('域名被拦截')) {
  shouldTerminateSequence = true  // 页面即将跳转/变化，后续工具在旧页面上无效
}
```

防止 AI 在一次 tool_calls 中同时调用 `navigate_to` + `click_element`，导致 click 在旧页面上执行。

#### 5.3.6 System Prompt 层面的行为约束（L4）

System Prompt 中嵌入了以下硬性规则：

| 规则 | 原文 |
|------|------|
| 强制自我评估 | "CALL工具前，必须在思考中评估上一轮操作是否成功（成功/失败/不确定）" |
| 验证操作结果 | "用 screenshot 或 read_page_content 验证操作结果，不要假设成功" |
| 失败切换策略 | "同一操作失败2-3次后必须换策略，严禁重复失败操作" |
| 系统页面边界 | "如果导航跳转到了系统页面(chrome://)，用finish_task立即汇报" |
| 跨步操作保护 | "禁止在navigate_to之后再调用其他DOM工具（页面已跳转）" |
| 跨站注入禁止 | "禁止跨站注入" |
| 能力边界声明 | "无匹配脚本 → finish_task('该网站暂无采集脚本')" |

## 十、关键设计决策

| 决策 | 说明 |
|------|------|
| **MV3 Service Worker** | 使用 module 类型，无持久化后台页面，适合间歇性 AI 请求 |
| **Port + storage 双通道** | SidePanel 与 Service Worker 通过 Port 流式通信，Agent 通过 storage 持久化写入历史（Plan B 架构：Port 断开不终止 Agent） |
| **Shadow DOM 隔离** | 所有页面注入的 UI 使用 closed Shadow DOM，避免样式污染和选择器冲突 |
| **Function Calling 降级** | AIService 检测到 400 错误时自动降级为不带 tools 的普通请求，兼容不支持 Function Calling 的模型 |
| **域名策略可选** | 策略未设置时全部放行，设置后才生效，降低配置门槛 |
| **iframe 侧边栏** | 浮动侧边栏通过 `<iframe>` 加载 `sidepanel.html`，复用同一套 UI 逻辑 |
| **软硬结合约束** | 第 3 层（Loop Detector / 预算通知）提供软提醒，改善失败后由第 1 层硬上限兜底终止 |
