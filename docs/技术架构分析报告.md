## 一、代码审查结果：特定场景限定问题

### 1.1 问题概述

作为通用的浏览器自动化程序，AI Browser 在代码中发现多处硬编码的特定场景关键词，违反了通用性原则。这些硬编码主要集中在业务意图识别、任务分类和模板命名中。

### 1.2 具体问题清单

#### 问题1：意图关键词硬编码（agent-runner.js）

**文件位置**：`chrome-extension/background/services/agent-runner.js:467-468`

**问题代码**：
```javascript
const INTENT_KEYWORDS = {
  '采集': ['采集', '批量'], '批量': ['批量', '采集'], '抓取': ['抓取', '采集'],
  '新闻': ['新闻', '采集'], '导出': ['导出', '下载'], '下载': ['下载', '导出'],
  '翻译': ['翻译'], '监控': ['监控'], '搜索': ['搜索'], '热点': ['热点', '热搜'],
}
```

**问题分析**：
- 将"采集"、"抓取"、"新闻"等特定业务词汇作为意图识别的关键词
- 这种硬编码会限制程序对其他场景的理解能力
- 例如："电商价格对比"、"学术文献检索"等场景无法被正确识别

**影响范围**：
- 自动搜索脚本时关键词扩展逻辑
- 页面内容关键词提取时的意图判断

#### 问题2：简单请求快速路径的特定场景判断（agent-runner.js）

**文件位置**：`chrome-extension/background/services/agent-runner.js:648/652`

**问题代码**：
```javascript
&& !userMessage.match(/采集|抓取|批量获取|爬|下载|打开|访问|点击/)
```

**问题分析**：
- 在判断用户请求是否为"简单数据请求"时，硬编码特定场景关键词
- 这种判断逻辑会导致其他类型的简单请求无法触发快速路径
- 例如："分析数据趋势"、"对比产品参数"等请求会被误判为复杂任务

#### 问题3：报告模板命名（report-templates.js）

**文件位置**：`chrome-extension/shared/report-templates.js:8-9`

**问题代码**：
```javascript
id: 'news_card_list',
name: '新闻卡片列表',
description: '适合新闻/文章类数据，每条一张卡片，含标题、链接、来源、时间、摘要',
```

**问题分析**：
- 模板ID和名称明确限定为"新闻"场景
- 描述中强调"适合新闻/文章类数据"，排斥其他数据类型
- 实际上这个模板的布局设计（卡片式展示）适用于任何列表数据展示

**实际影响**：
- 用户采集"产品列表"、"职位信息"、"学术论文"等数据时，可能不会选择此模板
- AI在自动推荐模板时，可能因为场景不匹配而错误排除此模板

#### 问题4：任务分类硬编码（task-template-service.js）

**文件位置**：`chrome-extension/background/services/task-template-service.js:20`

**问题代码**：
```javascript
const PREDEFINED_CATEGORIES = ['数据采集', '页面操作', '内容处理', '自动化', '其他']
```

**问题分析**：
- "数据采集"分类过于偏向特定场景，应该改为更通用的"数据处理"或"数据提取"
- 其他分类相对通用，但"数据采集"一词具有明显的场景倾向性

### 1.3 改进建议

#### 建议1：意图关键词改为动态扩展

**改进方案**：
```javascript
// 建议改为：根据用户消息和页面内容动态生成关键词
const userKeywords = extractKeywordsFromMessage(userMessage)
const pageKeywords = extractKeywordsFromPage(pageContent)
const expandedKeywords = expandKeywords([...userKeywords, ...pageKeywords])
```

**优点**：
- 支持任意场景的意图识别
- 根据上下文动态扩展关键词，提高准确性

#### 建议2：简单请求判断改为通用规则

**改进方案**：
```javascript
// 建议改为：基于任务复杂度的通用判断
const isSimpleRequest = 
  userMessage.length <= 20 &&  // 短消息
  hasStoredData(payloadSummary) &&  // 有历史数据
  !requiresNewPageAction(userMessage)  // 不需要新页面操作
```

**优点**：
- 不依赖特定场景关键词
- 基于客观指标判断任务复杂度

#### 建议3：模板命名改为通用化

**改进方案**：
```javascript
// 建议改为：
id: 'card_list',
name: '卡片列表',
description: '适合列表类数据，每条一张卡片，含标题、链接、来源、时间、摘要',
```

**优点**：
- 适用任何列表数据展示场景
- 不限定特定数据类型

#### 建议4：任务分类改为通用化

**改进方案**：
```javascript
// 建议改为：
const PREDEFINED_CATEGORIES = ['数据处理', '页面操作', '内容处理', '自动化', '其他']
```

**优点**：
- "数据处理"覆盖"数据采集"、"数据提取"、"数据分析"等多种场景
- 更通用、不限定特定业务

## 二、AI自主决策过程详解

### 2.1 决策流程架构

AI Browser 的自主决策过程采用**多层协作架构**，由以下核心组件共同完成：

```
┌─────────────────────────────────────────────┐
│         AgentService (入口层)               │
│  - 任务启动、状态管理、Port通信             │
└─────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────┐
│       AgentRunner (执行层)                  │
│  - 主循环控制、LLM调用、工具分发            │
└─────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────┐
│     TodoScheduler (调度层)                  │
│  - 待办进度管理、收敛控制、硬性规则         │
└─────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────┐
│       AgentJudge (评判层)                   │
│  - 结果自评、chatHistory存储、经验记录      │
└─────────────────────────────────────────────┘
```

## 三、多层上下文管理机制详解

### 3.1 四层上下文架构

AI Browser 采用**四层上下文管理架构**，解决传统对话系统的三大痛点：

| 痛点 | 传统方案 | AI Browser 解决方案 |
|------|---------|---------------------|
| 长任务上下文失忆 | 依赖 messages，清空后丢失 | WorkingMemory 独立存储，完整继承 |
| 大数据传输成本 | 全量数据发给AI | PayloadStore 暂存，只发摘要+引用 |
| 跨会话数据丢失 | 每次清空历史 | chatHistory持久化 + session隔离 |
| 缺乏进度反馈 | 无系统级进度管理 | TodoScheduler自动追踪+收敛提示 |

```
┌─────────────────────────────────────────────┐
│  Layer 1: WorkingMemory (工作记忆)         │
│  - 任务目标、页面状态、决策、发现           │
│  - 独立于对话流，AI可主动读写               │
└─────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────┐
│  Layer 2: PayloadStore (数据暂存)          │
│  - 工具结果大数据暂存，只发schema+样例      │
│  - 支持sessionId隔离 + 跨会话继承           │
└─────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────┐
│  Layer 3: chatHistory (历史对话)           │
│  - 持久化对话记录，支持连续对话             │
│  - 限制长度避免上下文过长                   │
└─────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────┐
│  Layer 4: TodoScheduler (待办调度)         │
│  - 扁平待办进度管理                         │
│  - 系统级收敛控制                           │
└─────────────────────────────────────────────┘
```

### 3.2 WorkingMemory：工作记忆层

#### 设计理念

**核心目标**：解决长任务中AI上下文失忆问题

**传统方案痛点**：
```javascript
// 传统方案：依赖 messages 数组
messages = [
  {role: 'user', content: '采集新闻列表'},
  {role: 'assistant', tool_calls: [...]},
  {role: 'tool', content: '结果...'},
  ...
]

// 问题：messages 清空后，AI无法回顾之前的决策和发现
// 例如：Round 10 时，AI忘记了 Round 3 的"排除分页采集，只取首页"决策
```

**AI Browser 方案**：
```javascript
// WorkingMemory：独立存储，不受 messages 清空影响
{
  taskGoal: "采集新闻列表页，内页也要采集",
  currentPage: {url: "...", title: "...", summary: "..."},
  discoveries: ["列表共15条", "页面有3个tab"],
  decisions: ["排除分页采集，只取首页"],
  excluded: ["选择器.xxx无结果"],
  dataRefs: [{key: "links", storeId: "p3", count: 15}]
}
```

#### 核心API

**初始化**（working-memory.js:35-46）：

export async function saveToChatHistoryStorage(content, toolCalls) {
  const historyData = await chrome.storage.local.get('chatHistory')
  const history = (historyData.chatHistory || []).slice()
  
  // 添加新记录
  const record = { role: 'assistant', content }
  if (toolCalls) {
    record.toolCalls = toolCalls.map(t => ({
      name: t.name,
      summary: String(t.result).slice(0, 200)
    }))
  }
  history.push(record)
  
  // 长度限制：最多50条，总字符8000
  const MAX_CHARS = 8000, MAX_ITEMS = 50
  let trimmed = history.slice(-MAX_ITEMS)
  
  // 从后向前保留，优先保留含附件的消息
  let totalChars = 0
  const keep = []
  for (let i = trimmed.length - 1; i >= 0; i--) {
    const m = trimmed[i]
    if (m.attachments) { keep.unshift(m); continue }
    totalChars += (m.content || '').length
    if (totalChars > MAX_CHARS && keep.length >= 2) break
    keep.unshift(m)
  }
  
  await chrome.storage.local.set({ chatHistory: keep })
}
