# Chrome Extension 功能迁移到 Electron (ai-browser) 工作量评估报告

**报告日期**: 2026-07-07  
**评估范围**: chrome-extension (Manifest V3) → ai-browser (Electron)  
**评估目的**: 分析技术可行性、工作量及迁移风险

---

## 1. 项目概述

### 1.1 Chrome Extension 架构概览

| 层级 | 文件数 | 核心模块 |
|------|--------|----------|
| Background Service Worker | 27个服务 | AgentService, AIService, ToolService, ScriptService, ConfigService, DBService, TodoScheduler, WorkingMemory, ContextCompressor 等 |
| SidePanel UI | 6个文件 | sidepanel.js (2700+行), conversation-viewer.js, todo-viewer.js, feature-panels.js |
| Content Script | 2个文件 | index.js (900+行), network-capture.js |
| Shared Utilities | 4个文件 | utils.js, export-service.js, report-templates.js, template-engine.js |

**关键特性**:
- Manifest V3 Service Worker 架构
- AI Agent 自主决策循环 (Agent Runner + Tool Builder)
- 多层上下文管理 (Working Memory + Context Compressor + Scratchpad)
- 动态脚本注入系统 (Script Service + Tool Recording)
- 定时任务调度 (Scheduled Task Service + Todo Scheduler)

### 1.2 Electron (ai-browser) 现有架构

| 层级 | 文件数 | 核心模块 |
|------|--------|----------|
| Main Process | 8个文件 | main.js (1025行), tab_manager.js, preload.js |
| AI Engine | 9个文件 | agent_loop.js, llm_provider.js, tool_executor.js, tool_definitions.js, action_executor.js, analyzer.js, electron_engine.js, uncertainty_guard.js |
| Admin Server | 完整后端 | Express API + SQLite + 文件上传 |

**已实现功能**:
- BrowserView 多标签页管理
- AI 对话 (流式 + Function Calling)
- 统一工具调用架构 (26个工具定义)
- 逆向分析 (网络请求捕获 + JS脚本提取)
- 智能操作 (ActionExecutor + AgentLoop)
- 管理后台 API 集成

## 3. Chrome API vs Electron API 对照

### 3.1 核心API映射

| Chrome Extension API | Electron API | 可替代性 | 说明 |
|----------------------|--------------|----------|------|
| `chrome.runtime.sendMessage` | `ipcRenderer.invoke` / `ipcMain.handle` | ✅ 完全替代 | IPC通信机制更强大 |
| `chrome.runtime.connect` (Port) | `ipcRenderer.on` + EventEmitter | ⚠️ 需重构 | 流式通信改用事件监听 |
| `chrome.storage.local` | `electron-store` / `fs` | ✅ 完全替代 | 文件存储更灵活 |
| `chrome.tabs.query/update` | `BrowserView` API | ✅ 完全替代 | 直接控制视图 |
| `chrome.scripting.executeScript` | `webContents.executeJavaScript` | ✅ 完全替代 | 执行JS能力更强 |
| `chrome.sidePanel` | 自定义BrowserView | ✅ 替代方案 | 不依赖原生sidePanel |
| `chrome.contextMenus` | Electron Menu API | ✅ 完全替代 | 菜单API更灵活 |
| `chrome.alarms` | Node.js timers / cron | ✅ 完全替代 | 定时器不受SW限制 |
| `chrome.webRequest` | `session.webRequest` | ✅ 完全替代 | 网络拦截能力相同 |
| `chrome.notifications` | Electron Notification | ✅ 完全替代 | 原生通知 |
| Service Worker生命周期 | Main Process持久运行 | ✅ 优势 | 无30秒休眠限制 |

### 3.2 关键差异

| 特性 | Chrome Extension | Electron | 迁移影响 |
|------|------------------|----------|----------|
| **生命周期** | Service Worker 30秒休眠 | 持久运行 | ✅ 消除状态丢失问题 |
| **DOM访问** | 无法直接访问 | webContents可直接操作 | ✅ 更强大的页面操作能力 |
| **文件系统** | 受限 | 完全访问 | ✅ 支持本地文件读写 |
| **原生模块** | 受限 | 可加载Node模块 | ✅ 支持原生库集成 |
| **跨域请求** | 需host_permissions | 无限制 | ✅ 更灵活的网络访问 |
| **分发方式** | Chrome商店审核 | 自由分发 | ✅ 无审核限制 |

## 5. 迁移难点与技术风险

### 5.1 高风险项

| 风险项 | 风险等级 | 影响范围 | 缓解措施 |
|--------|----------|----------|----------|
| **Service Worker → Main Process 状态迁移** | 🔴 高 | Agent状态、会话管理 | 设计状态持久化层，支持进程重启恢复 |
| **Port流式通信重构** | 🔴 高 | AI对话、Agent实时反馈 | 使用EventEmitter+IPC实现双向流 |
| **Content Script注入机制差异** | 🔴 高 | 划词工具、页面助手 | BrowserView preload + executeJavaScript替代 |
| **SidePanel UI架构差异** | 🟡 中 | 整体交互体验 | 设计可复用的UI组件库 |
| **Agent循环中断恢复** | 🟡 中 | 长任务执行 | 实现任务快照+断点续传 |

### 5.2 中风险项

| 风险项 | 风险等级 | 影响范围 | 缓解措施 |
|--------|----------|----------|----------|
| **Storage数据格式差异** | 🟡 中 | 配置迁移 | 设计兼容层，渐进迁移 |
| **定时任务调度精度** | 🟡 中 | ScheduledTaskService | 使用node-cron替代chrome.alarms |
| **网络请求拦截API差异** | 🟡 中 | Analyzer | session.webRequest行为验证 |
| **UI样式一致性** | 🟢 低 | 用户体验 | 统一CSS变量+主题系统 |

### 5.3 技术债务

| 问题 | 来源 | 建议 |
|------|------|------|
| SidePanel代码量大(2700+行) | Chrome Extension | 模块化拆分，按功能分离 |
| AgentService复杂度高 | Chrome Extension | 简化状态管理，统一AgentRunner |
| 多层上下文管理混乱 | 两边都有 | 设计统一的ContextManager类 |

## 7. 迁移收益分析

### 7.1 技术收益

| 收益项 | 说明 |
|--------|------|
| **消除Service Worker生命周期限制** | 不再受30秒休眠影响，Agent可长时间运行 |
| **更强大的页面操作能力** | BrowserView可直接操作DOM，无需content script转发 |
| **原生文件系统访问** | 支持本地文件读写、导出、持久化 |
| **无审核限制** | 自由分发，无Chrome商店审核周期 |
| **更灵活的网络访问** | 无跨域限制，支持任意API调用 |
| **Node.js生态集成** | 可使用任意npm包，扩展能力强 |

### 7.2 产品收益

| 收益项 | 说明 |
|--------|------|
| **桌面应用体验** | 更稳定的桌面应用，不受浏览器更新影响 |

## 8. 建议与结论

### 8.1 核心建议

1. **优先完成基础架构迁移**: IPC和Storage是后续工作的基础
2. **Agent核心优先于UI**: 确保Agent能力完整后再迁移UI
3. **渐进式迁移**: 不建议一次性替换，建议两版本并行过渡
4. **代码模块化**: 迁移时重构，避免复制大文件
5. **状态持久化设计**: 重点关注Agent状态恢复机制

### 8.2 结论

| 维度 | 评估结果 |
|------|----------|
| **技术可行性** | ✅ 完全可行，Electron API可完全替代Chrome API |
| **工作量** | 约54人天(2.7人月)，中等规模 |
| **风险等级** | 🟡 中等，主要风险在Port重构和Content Script迁移 |
| **ROI** | ✅ 高收益，解决SW生命周期问题并获得更强能力 |
| **建议启动时间** | 建议2-3人并行开发，周期约2个月 |
