# AI Browser 脚本编写要求

## 一、脚本格式

### 1.1 单文件脚本

上传一个 `.js` 文件，系统自动将其作为一条模块记录（`load_order = 0`）。

```javascript
// @name: 我的脚本
// @description: 脚本功能描述
// @version: 1.0.0
// @urlPattern: *://example.com/*

(function() {
  const params = window.__SCRIPT_PARAMS__ || {};
  // 脚本逻辑
})();
```

### 1.2 多模块脚本

上传多个 `.js` 文件，系统按 **文件名字典序** 自动分配 `load_order`（0, 1, 2...）。

**命名规范**：用数字前缀控制加载顺序：

```
00-config.js      → load_order: 0  （最先加载）
01-style.js       → load_order: 1
02-utils.js       → load_order: 2
03-fieldReader.js → load_order: 3
...
09-content.js     → load_order: 9  （最后加载）
```

> 前端上传时会按文件名排序后依次分配 `load_order`，最终注入时按 `load_order ASC` 拼接所有模块代码。

### 1.3 元数据注释

脚本文件开头可写 `// @key: value` 格式的注释，上传时自动识别：

| key | 说明 | 默认值 |
|-----|------|--------|
| `@name` | 脚本名称 | 文件名（去 .js） |
| `@description` | 脚本描述 | 空 |
| `@version` | 版本号 | `1.0.0` |
| `@urlPattern` | URL 匹配模式 | `*` |

## 二、参数系统

### 2.1 参数定义（params_schema）

在管理后台"脚本参数"区域添加，每个参数包含：

| 字段 | 必填 | 说明 |
|------|------|------|
| `key` | 是 | 参数键名，脚本中通过此键读取值 |
| `label` | 是 | 前端显示标签 |
| `type` | 是 | 参数类型：`text` / `select` / `number` / `boolean` |
| `options` | 否 | 选项数组，仅 `type=select` 时需要 |
| `default` | 否 | 默认值 |

**示例**：

| key | label | type | options | default |
|-----|-------|------|---------|---------|
| providerMode | AI来源 | select | mock,coze,dmx | mock |
| apiUrl | 接口地址 | text | | https://example.com/api |
| maxRetries | 重试次数 | number | | 3 |
| debugMode | 调试模式 | boolean | | false |

### 2.2 参数赋值（params_data）

在管理后台填写每个参数的实际值，优先级：**params_data 中的值 > params_schema 中的 default**。

### 2.3 脚本中读取参数

注入时系统会在代码前自动注入：

```javascript
window.__SCRIPT_PARAMS__ = { "providerMode": "coze", "apiUrl": "https://..." };
```

脚本中读取：

```javascript
const params = window.__SCRIPT_PARAMS__ || {};
const mode = params.providerMode || 'mock';
```

## 三、URL 匹配模式

| 模式 | 含义 |
|------|------|
| `*` | 匹配所有页面 |
| `*://example.com/*` | 匹配 example.com 的所有页面（http/https） |
| `http://work.66wz.com:7001/pub/*` | 匹配指定端口和路径 |
| 多个模式用逗号分隔 | 如 `*://a.com/*, *://b.com/*` |

`*://` 同时匹配 `http` 和 `https`。

## 四、模块编写规范

### 4.1 配置模块（最先加载）

负责读取参数并构建全局配置对象：

```javascript
// 00-config.js
(function() {
  const params = window.__SCRIPT_PARAMS__ || {};
  window.MY_CONFIG = {
    mode: params.mode || 'default',
    apiUrl: params.apiUrl || '',
  };
})();
```

### 4.2 功能模块

直接定义全局对象或函数，供后续模块调用：

```javascript
// 02-utils.js
const MyUtils = {
  formatDate(date) { ... },
  debounce(fn, ms) { ... },
};
```

### 4.3 入口模块（最后加载）

负责初始化和事件绑定：

```javascript
// 09-content.js
if (window.__MY_SCRIPT_LOADED__) return;
window.__MY_SCRIPT_LOADED__ = true;

document.addEventListener('DOMContentLoaded', function() {
  // 初始化逻辑
});
```

### 4.4 防重复注入

入口模块应加防重复注入检查：

```javascript
if (window.__MY_SCRIPT_LOADED__) return;
window.__MY_SCRIPT_LOADED__ = true;
```

## 五、上传流程

1. 在管理后台点击"上传脚本"
2. 填写脚本名称、分类（必填）、URL 匹配模式
3. 选择一个或多个 `.js` 文件
4. 点击上传，系统自动：
   - 单文件：作为一条模块存储
   - 多文件：按文件名排序分配 `load_order`，每条文件作为一条模块存储
5. 上传后在脚本编辑页面配置参数

## 六、更新脚本代码

修改脚本代码有两种方式：

1. **管理后台编辑**：打开脚本详情 → 模块管理 → 编辑模块代码 → 保存
2. **数据库直接更新**：修改本地文件后，用脚本将新代码写入 `script_modules` 表

> 注意：修改本地 `.js` 文件不会自动更新数据库中已存储的模块代码。客户端每次加载页面时从数据库实时拉取代码。
