# 架构升级与专利分析记录

## 一、双引擎架构改造

### 背景

原有系统采用纯 JS 注入方式控制浏览器，存在以下问题：
- `document.querySelector().click()` 不触发完整事件链（mousedown→focus→mouseup→click）
- 现代网站依赖中间事件，导致交互失败率高
- 代码执行结果不确定，可能触发页面导航、DOM 变更、JS 运行错误

### 方案选择

对比了三种方案后选择 **方案 A：纯 Electron API**

| 方案 | 说明 | 选择 |
|------|------|------|
| A: 纯 Electron API | sendInputEvent + executeJavaScript，无额外依赖 | ✅ |
| B: Electron + CDP | debugger API 调用 CDP 命令，与 DevTools 冲突 | ❌ |
| C: 集成 Browser-use | Python 子进程，架构复杂 | ❌ |

### 核心发现

Electron 的 `webContents` 已提供 CDP 级别能力：
- `sendInputEvent()` = CDP `Input.dispatchMouseEvent`（完整鼠标/键盘事件链）
- `insertText()` = 可靠文字输入（触发 input 事件）
- `executeJavaScript()` = CDP `Runtime.evaluate`（且不受 CSP 限制）

### 改造内容

#### 新增文件

1. **electron/ai/electron_engine.js** — 引擎1：Electron API 封装
   - `clickElement()`: sendInputEvent 完整事件链（mouseMove→mouseDown→mouseUp）
   - `typeText()`: insertText + 焦点 + 清空 + change 事件
   - `pressKey()`: 键盘事件（keyDown→keyUp）
   - `getElementRect()`: 获取元素位置并滚动到可视区
   - `isElementVisible()`: 检查元素可见性

2. **electron/ai/uncertainty_guard.js** — 不确定性检测模块
   - `captureDomSnapshot()`: DOM 快照（元素计数 + 类型统计 + 内容哈希）
   - `detectDomChanges()`: 对比快照检测变更
   - `checkLoop()`: 代码哈希 + 重复计数（3次警告/5次强制）
   - `verifyOperation()`: 操作结果验证（点击后检查 DOM 变化、输入后验证值）
   - `getLoopGuidance()`: 循环检测引导 prompt 生成

#### 改造文件

3. **electron/ai/tool_executor.js** — 完全重写为双引擎架构
   - click_element → 引擎1（sendInputEvent）
   - type_text → 引擎1（insertText）【新增工具】
   - press_key → 引擎1（sendInputEvent 键盘）【新增工具】
   - execute_js → 引擎2（JS 注入 + DOM 变更感知 + 循环检测）
   - 所有工具统一加入不确定性检测

4. **electron/ai/tool_definitions.js** — 新增工具定义
   - click_element 描述更新（强调 Electron 原生事件）
   - 新增 type_text 工具定义
   - 新增 press_key 工具定义
   - 工具总数：14 → 17

5. **electron/ai/action_executor.js** — executeInPage 增强
   - 新增 `_captureDomFingerprint()` 方法
   - executeInPage 执行前后拍 DOM 指纹快照
   - 返回结果包含 `domChanged` 和 `domChanges` 信息

6. **electron/ai/agent_loop.js** — 智能体循环增强
   - 新增 `actionHashes` 循环检测历史
   - 新增 `_checkLoop()` 方法（代码哈希 + 重复计数）
   - 反馈消息附加 DOM 变更/循环/验证提示
   - 系统提示更新：告知 AI 系统能力（DOM 检测、循环检测、操作验证）

7. **electron/main.js** — 工具结果传递增强
   - unified:tool-result 事件增加 loopWarning 字段
   - 工具结果消息附加循环检测警告到 AI 上下文

### 解决的不确定性问题

| # | 问题 | 解决方式 |
|---|------|---------|
| 1 | AI 错误自愈引导 | 循环检测时注入引导 prompt |
| 2 | 循环检测 | 代码哈希+重复计数，3次警告/5次强制换方法 |
| 3 | DOM 变更感知 | 执行前后 DOM 指纹快照对比 |
| 4 | click 不完整事件链 | sendInputEvent: mouseMove→mouseDown→mouseUp |
| 5 | 输入不可靠 | insertText + change 事件触发 |
| 6 | 操作结果验证 | 点击后检查 DOM 变化，输入后验证值 |
| 7 | 反馈消息增强 | DOM变更/循环/验证提示注入 AI 上下文 |
| 8 | 导航后上下文重建 | 已有机制保留（URL 对比 + waitForPageLoad） |

## 二、与 Browser-use 的深度对比

### 架构差异

| 维度 | Browser-use | 本系统 |
|------|-------------|---------|
| 浏览器控制 | CDP (Chrome DevTools Protocol) | Electron BrowserView |
| AI 角色 | 指挥官（选择预定义动作） | 程序员（生成任意 JS 代码） |
| 能力边界 | 受限于预定义动作数量 | 理论上无上限 |
| 控制方式 | API 级，结果确定 | 代码注入，结果不确定 |
| 页面导航感知 | CDP 生命周期事件（主动） | URL 前后对比（被动发现） |
| 思维链 | 内置 thinking/evaluation/memory | 无 |
| 循环检测 | 动作哈希+页面指纹 | 代码哈希+重复计数 |
| 工具协议 | 结构化 JSON 输出 | OpenAI Function Calling |
| 返回值限制 | 20K 字符截断 | 30K 且可配置 |

### 控制范式差异

```
Browser-use：AI 是"指挥官"，下指令，CDP API 是"执行兵"，按指令动作
本系统：  AI 是"程序员"，写代码，浏览器是"运行环境"，执行代码
```

### Browser-use 放弃 JS 注入的原因

| 原因 | 说明 |
|------|------|
| JS .click() 不触发完整事件链 | CDP 鼠标事件更可靠 |
| DOM 结构不可靠 | 每站不同，CDP 坐标点击不需要 DOM |
| CSP 拦截 | CDP 不受 CSP 限制 |
| 异步状态不确定 | CDP 有内置等待机制 |

### 本系统的应对方案

| Browser-use 放弃的原因 | 本系统解决方案 |
|------|------|
| JS .click() 不完整 | sendInputEvent 完整事件链 |
| DOM 不可靠 | collectPageContext 先感知再操作 |
| CSP 拦截 | Electron 不受 CSP 限制 |
| 异步不确定 | URL 变化检测 + 等待加载 + 上下文重建 |

### 宿主 vs 观察者（核心差异）

```
Browser-use：浏览器"外部观察者"，通过 CDP 协议远程操控
本系统：    浏览器"宿主进程"，拥有内核级访问权限
```

| 能力 | Browser-use（外部观察者） | 本系统（宿主） |
|------|--------------------------|--------------|
| CSP 限制 | 受限 | 不受限制 |
| 完整页面保存 | 无 | savePage() |
| 请求拦截 | CDP 可实现 | 已有 |
| 文件系统 | 无 | 完整 Node.js |
| 窗口控制 | 无 | BrowserWindow |
| 注入 UI 面板被 AI 感知 | ❌ 盲区 | ✅ 出现在 DOM 摘要中 |

## 三、专利创新点分析

### 最终确定的创新点

**1 个核心创新 = AI 代码生成注入执行中的上下文一致性保障方法**

### 技术问题

AI 生成的代码注入浏览器执行后，可能触发：
1. 页面导航 → 后续 JS 在已销毁页面上执行 → 失败
2. DOM 变更 → AI 的页面上下文与实际不一致 → 决策错误
3. JS 执行错误 → 代码逻辑有误 → 需要重新生成

### 技术方案

```
步骤1：AI 生成 JS 代码前，收集页面上下文快照
步骤2：AI 基于上下文生成 JS 代码，注入执行
步骤3：执行后自动检测三类异常
步骤4：根据异常类型采取恢复策略
步骤5：恢复后的最新页面上下文注入 AI 对话
步骤6：循环直至任务完成
```

### 创新性论证

| 论据 | 说明 |
|------|------|
| 现有技术回避了该问题 | Browser-use 有意放弃 JS 注入，选择预定义动作 |
| 本系统直面该问题 | 不仅用 JS 注入，还解决了不确定性 |
| 不同控制范式带来不同技术问题 | 代码注入 ≠ API 控制，需要全新机制 |
| "宿主"vs"观察者"架构差异 | Electron 拥有内核级权限，CDP 方案无法达到 |

### 权利要求框架

```
独立权利要求（方法）：
  S1. 接收自然语言任务
  S2. 收集当前浏览器页面的上下文信息
  S3. 大语言模型基于上下文生成 JavaScript 代码
  S4. 将代码注入浏览器页面执行
  S5. 检测执行后页面状态变化（URL变更/JS错误/加载状态）
  S6. 若检测到异常，执行恢复策略
  S7. 将最新页面上下文和执行结果反馈大语言模型
  S8. 循环 S3-S7 直至模型判定任务完成

从属权利要求：
  2. S3中模型同时可选择调用预定义浏览器工具
  3. S5中URL变更检测方法
  4. S6中等待加载的方法
  5. S6中错误恢复策略
  6. S2中上下文包含的信息
  7. 预定义工具包括的类型
  8. 上下文窗口管理方法
```

## 四、Chrome Extension 迁移可行性分析

### 三种方案对比

| 维度 | Electron（当前） | 插件+本地服务 | 插件+云端服务 |
|------|----------------|-------------|-------------|
| 功能完整度 | 100% | ~90% | ~50-60% |
| sendInputEvent | ✅ | ✅(CDP) | ❌ |
| 请求拦截 | ✅ | ✅(CDP) | ❌ |
| 文件系统 | ✅ | ✅ | ❌ |
| CSP 限制 | 无 | 无 | 有 |
| 离线使用 | ✅ | ✅ | ❌ |
| 分发难度 | 低 | 高 | 中 |

### 结论

保持 Electron 架构。"浏览器宿主"定位是核心技术壁垒和专利差异化。
